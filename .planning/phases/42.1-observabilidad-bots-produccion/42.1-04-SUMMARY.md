# Phase 42.1 Plan 04: Anthropic Fetch Capture + Prompt Versioning Summary

**One-liner:** Closes Wave 2 by completing the Anthropic side of `makeObservableFetch` — request body parsing, response token/content extraction, deterministic prompt hashing, and a layered `purposeAls` so call sites can tag the logical reason for each Claude call without threading params through the SDK.

## What was built

### Task 1 — `prompt-version.ts` + `anthropic-instrumented.ts`

Two new leaf modules:

- **`prompt-version.ts`** owns deterministic prompt hashing and the upsert helper that the eventual `flush()` (Plan 07) calls against `agent_prompt_versions`. Two exports:
  - `hashPrompt(systemPrompt, model, params)` — SHA-256 of `JSON.stringify({p, m, t, x})` with whitespace normalised (`.trim().replace(/\s+/g, ' ')`) and a fixed key order. Same prompt + same params produces the same hash regardless of caller key order.
  - `resolvePromptVersions(supabase, prompts)` — batch upsert via `.upsert(rows, { onConflict: 'prompt_hash', ignoreDuplicates: false }).select('id, prompt_hash')` returning a `Map<promptHash, id>`. Caller passes a `createRawAdminClient()` to avoid recursing through the instrumented admin wrapper.

- **`anthropic-instrumented.ts`** exposes `createInstrumentedAnthropic(opts?)`. This is the SINGLE place in the repo where `new Anthropic({...})` is allowed (the lint-able invariant Plans 05/06 enforce). It injects `makeObservableFetch(fetch, 'anthropic')` cast to `typeof fetch` at the SDK boundary.

```ts
export interface CreateInstrumentedAnthropicOpts {
  apiKey?: string
  baseURL?: string
}
export function createInstrumentedAnthropic(
  opts: CreateInstrumentedAnthropicOpts = {},
): Anthropic
```

### Task 2 — Anthropic capture branch + `purposeAls`

`fetch-wrapper.ts` now fully implements the `kind === 'anthropic'` path:

- **Request body parsing.** `parseAnthropicRequestBody` defensively `JSON.parse`s `init.body` (the SDK always sends a JSON string). Returns `null` on any failure — wrapper never throws.
- **System prompt normalisation.** `normalizeSystemPrompt` accepts both the string form and the content-block array form (used for prompt caching) and joins all `{type:'text', text}` blocks with newlines. The flat string is what gets hashed.
- **Streaming detection.** `isStreamingResponse` returns true if `requestBodyParsed.stream === true` OR the response `content-type` contains `text/event-stream`. In that case the wrapper records a coarse `ai_call_streaming` event and returns the response untouched. Verified in research that production agents do not currently use streaming, so this fast-path costs nothing today but keeps the wrapper safe if streaming is introduced later.
- **Non-streaming capture.** `response.clone().json().catch(() => null)` extracts an `AnthropicResponseBody` shape and feeds `recordAiCall` with: `purpose`, `systemPrompt`, `model`, `temperature`, `maxTokens`, `messages`, `responseContent` (raw `content` blocks), `inputTokens`, `outputTokens`, `cacheCreationInputTokens`, `cacheReadInputTokens`, `durationMs`, `statusCode`, and on HTTP error a stringified `error` derived from `responseBody.error`.

`collector.ts` had its `hashPromptStub` deleted and now imports the real `hashPrompt` from `./prompt-version`. `recordAiCall` computes the real hash inside the existing try/catch so it remains throw-safe.

### Decision recorded — `purposeAls` (second AsyncLocalStorage)

Two ALS layers, each with its own lifetime:

| ALS                | Scope                                | Why separate                                                                 |
| ------------------ | ------------------------------------ | ---------------------------------------------------------------------------- |
| collector ALS      | Whole turn (Inngest handler invocation) | Lifetime tied to flush() at end of turn                                       |
| purposeAls         | Single `messages.create(...)` call     | Caller-controlled label of which pipeline step issued the call               |

Plans 05/06 will wrap each call site as:

```ts
await runWithPurpose('comprehension', () =>
  anthropic.messages.create({...})
)
```

When `runWithPurpose` is not used, `getCurrentPurpose()` returns null and the wrapper records `purpose: 'unknown'`. Plan 11 reviews the unknown-purpose ratio in production telemetry to decide whether more migration work is required.

Both helpers are re-exported from `index.ts` along with `createInstrumentedAnthropic`, `hashPrompt`, `resolvePromptVersions`, and their supporting types.

## Decisions Made

1. **`purposeAls` is a SECOND AsyncLocalStorage**, not a header on the request body or a `metadata` field. Rationale: layered ALS keeps the collector ALS lifetime stable (full turn) while letting purpose lifetime be as narrow as a single `messages.create()` call. Avoids polluting the Anthropic request payload with internal labels.
2. **Streaming responses are not consumed.** Verified in research that production agents do not stream; the wrapper records a coarse `ai_call_streaming` event and returns the response untouched so the SDK's stream parser keeps working when streaming is added later.
3. **`createInstrumentedAnthropic` is the only construction point.** Plans 05/06 will migrate the 13 existing call sites; afterwards a CI grep enforces the invariant.
4. **`hashPrompt` whitespace normalisation is permissive.** Cosmetic re-indents do not generate new prompt versions, which is the desired dedup behaviour for the `agent_prompt_versions` table.
