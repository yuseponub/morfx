  <behavior>
    The composeBlock function takes:
    - newByIntent: Map<string, PrioritizedTemplate[]> — templates from current orchestrator, grouped by intent
    - pending: PrioritizedTemplate[] — templates saved from previous interrupted block
    - maxBlockSize: number (default 3) — max templates per block

    Returns BlockCompositionResult:
    - block: PrioritizedTemplate[] — max 3 templates to send
    - pending: PrioritizedTemplate[] — overflow that becomes new pending (CORE/COMP only)
    - dropped: PrioritizedTemplate[] — OPC that don't fit (permanently discarded)

    Algorithm (from CONTEXT.md locked decisions):
    1. Intent cap: Max 3 intents. If 4+ intents, excess intents (all their templates) go to pending immediately.
    2. For each selected intent (max 3): take the CORE template (orden=0 / lowest orden).
    3. CORE from NEW intents go into block first (respond to what client just asked).
    4. Remaining templates (non-CORE from selected intents + all pending) go into a pool.
    5. Deduplication: same template_id in both new and pending -> keep one copy (prefer pending at same priority).
    6. Sort pool by priority rank (CORE=0 > COMP=1 > OPC=2), tiebreaker: pending wins over new.
    7. Fill block up to cap from sorted pool.
    8. Overflow: OPC that don't fit -> dropped (permanent). CORE/COMP that don't fit -> pending (next cycle).

    Types:
    - TemplatePriority = 'CORE' | 'COMPLEMENTARIA' | 'OPCIONAL'
    - PRIORITY_RANK: Record<TemplatePriority, number> = { CORE: 0, COMPLEMENTARIA: 1, OPCIONAL: 2 }
    - PrioritizedTemplate = { templateId: string, content: string, contentType: 'texto' | 'template' | 'imagen', priority: TemplatePriority, intent: string, orden: number, isNew: boolean }
    - BlockCompositionResult = { block: PrioritizedTemplate[], pending: PrioritizedTemplate[], dropped: PrioritizedTemplate[] }

    Test cases (input -> expected output):

    Case 1: Single intent, 1 template
      new: { precio: [CORE] }, pending: []
      -> block: [precio-CORE], pending: [], dropped: []

    Case 2: Single intent, 5 templates (1 CORE, 2 COMP, 2 OPC)
      new: { precio: [CORE, COMP1, COMP2, OPC1, OPC2] }, pending: []
      -> block: [CORE, COMP1, COMP2] (3 = cap), pending: [], dropped: [OPC1, OPC2]

    Case 3: 3 intents, 1 CORE each = cap exactly
      new: { precio: [CORE], envio: [CORE], pago: [CORE] }, pending: []
      -> block: [precio-CORE, envio-CORE, pago-CORE], pending: [], dropped: []

    Case 4: 3 intents with secondary templates -> secondary goes to pending
      new: { precio: [CORE, COMP], envio: [CORE, COMP], pago: [CORE] }, pending: []
      -> block: [precio-CORE, envio-CORE, pago-CORE] (cap 3), pending: [precio-COMP, envio-COMP], dropped: []

    Case 5: 4 intents -> excess intent goes to pending
      new: { precio: [CORE], envio: [CORE], pago: [CORE], garantia: [CORE, COMP] }, pending: []
      -> block: [precio-CORE, envio-CORE, pago-CORE], pending: [garantia-CORE, garantia-COMP], dropped: []

    Case 6: Pending merge — pending CORE displaces new OPC
      new: { precio: [CORE, OPC] }, pending: [{ envio-CORE, isNew: false }]
      -> block: [precio-CORE, envio-CORE-pending, precio-OPC...wait, cap=3]
      Actually: block has precio-CORE (new intent first), then fill: envio-CORE-pending (rank 0), precio-OPC (rank 2)
      -> block: [precio-CORE, envio-CORE-pending, precio-OPC], pending: [], dropped: []

    Case 7: Pending merge at cap — pending CORE takes slot, OPC dropped
      new: { precio: [CORE, OPC1, OPC2] }, pending: [{ envio-COMP, isNew: false }]
      -> block: [precio-CORE, envio-COMP-pending, precio-OPC1], pending: [], dropped: [precio-OPC2]

    Case 8: Deduplication — same template_id in new and pending
      new: { precio: [CORE(id=t1)] }, pending: [{ id=t1, CORE, isNew: false }]
      -> block: [t1] (only once), pending: [], dropped: []

    Case 9: Tiebreaker — pending wins over new at same priority
      new: { precio: [COMP(id=t1)] }, pending: [{ id=t2, COMP, isNew: false }]
      Pool after CORE selection: [t2-COMP-pending, t1-COMP-new] (pending first)
      -> block: [t2-pending, t1-new] if space

    Case 10: Empty input
      new: empty map, pending: []
      -> block: [], pending: [], dropped: []

    Case 11: Only pending, no new
      new: empty map, pending: [CORE, COMP, OPC, OPC2]
      -> block: [CORE, COMP, OPC], pending: [], dropped: [OPC2]
