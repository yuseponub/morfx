<objective>
Close GAP-41-09: an audio-only `.mp4`/`.mov` sent to Instagram or Facebook delivers ONCE (DB has a single outbound row, status=sent, type=audio) but leaves a phantom greyed bubble stuck at status='sending' that never reconciles.

Root cause (VERIFIED in code): the realtime INSERT reconciler in `src/hooks/use-messages.ts:382-396` matches the optimistic media bubble to the real row by `msg.type === newMessage.type && caption === caption`. The optimistic type is computed in the composer as `deriveMediaType(file.type)` where a `.mp4`'s `file.type` is `'video/mp4'` → optimistic type = `'video'`. GAP-41-13 reclassifies the SERVER row to `type='audio'` for audio-only mp4. So real `'audio'` != optimistic `'video'` → `optimisticIndex === -1` → the real row is appended AND the optimistic stays forever at `status='sending'`.

Fix: align the CLIENT optimistic classification with the server. Add a browser-safe `isAudioOnlyMp4Bytes(Uint8Array)` detector and apply it in the composer (gated to IG/FB + `video/mp4|video/quicktime` + audio-only) so the optimistic type becomes `'audio'` and the reconciler swaps in place. The reconciler itself is NOT touched (we fix the INPUT, not the matcher). WhatsApp is NOT touched.

Purpose: outbound media optimistic-UI correctness (IG-02 UX) — eliminate the phantom clone without loosening the reconciler or risking false matches.
Output: a refactored pure detector module + a composer override + unit tests, pushed to origin/main.
</objective>

```typescript
export function isAudioOnlyMp4(buf: Buffer): boolean {
  try {
    if (!Buffer.isBuffer(buf) || buf.length < 8) return false
    const slice = buf.length > 524288 ? buf.subarray(0, 524288) : buf
    const hasVide = slice.indexOf('vide', 0, 'ascii') !== -1
    if (hasVide) return false
    const hasSoun = slice.indexOf('soun', 0, 'ascii') !== -1
    return hasSoun
  } catch {
    return false
  }
}
```
