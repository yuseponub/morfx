# Phase 44.1 — CRM bots config via `platform_config` table

## Why this phase exists

Post-Phase-44 QA checkpoint (Plan 44-09 Task 6): user pushed back on storing 3 non-secret config vars (`CRM_BOT_ENABLED`, `CRM_BOT_RATE_LIMIT_PER_MIN`, `CRM_BOT_ALERT_FROM`) in Vercel env vars. Reasons:
1. User operates "VARIAS cuentas con bots independientes" (multi-workspace) and doesn't want to "llenar Vercel" with ops vars.
2. Wants kill-switch flipable without Vercel redeploy (Blocker 6 of Phase 44 documented that env refresh requires a deployment cycle — user wants to eliminate that).
3. Prefers centralizing bot config in DB for future per-workspace overrides and admin UI.

`RESEND_API_KEY` stays as Vercel env var — it's a secret and must be secret-managed.

## Decisions locked (from conversation 2026-04-18/19)

- **D1:** New table `platform_config(key TEXT PK, value JSONB NOT NULL, updated_at TIMESTAMPTZ)` in public schema. Additive migration. No RLS (same rationale as `crm_bot_actions` — server-only via `createAdminClient()`, filtered server-side; RESEARCH §"Deprecated" + Phase 44 pattern).
- **D2:** 3 keys seeded initially: `crm_bot_enabled` (bool), `crm_bot_rate_limit_per_min` (int), `crm_bot_alert_from` (string or null for Resend sandbox fallback).
- **D3:** Values stored as JSONB (typed JSON values — booleans, numbers, strings natively). Reader coerces via discriminated union (`getPlatformConfig<T>(key, fallback)`).
- **D4:** In-memory cache with **30s TTL** (not a process-wide singleton with manual invalidation — simple time-based expiry). Cold read on first call post-restart or post-expiry. Multi-instance consistency: eventual within 30s.
- **D5:** Reader returns a **safe fallback** if DB read fails (DB down, row missing, parse error). Fallback matrix:
  - `crm_bot_enabled` → `true` (fail-open for reads, fail-closed for writes — details in plan)
  - `crm_bot_rate_limit_per_min` → `50`
  - `crm_bot_alert_from` → `undefined` (which causes alerts.ts to use Resend sandbox `onboarding@resend.dev` per its existing behavior)
- **D6:** Refactor ONLY the 4 existing consumer files (3 routes + alerts.ts). NO changes to the executor, tools, or domain layer. Tools never read config — they're invoked by the routes/alerts which gate on config.
- **D7:** No admin UI in this phase. User edits values via Supabase Studio SQL Editor for now. Admin UI is a follow-up phase.
- **D8:** No per-workspace overrides in this phase. Column `workspace_id UUID NULL` NOT added yet — schema stays minimal. Future phase can add it non-breaking.
- **D9:** `CRM_BOT_RATE_LIMIT_PER_MIN` env var is currently read by `rate-limiter.ts` DEFAULTS record in the form `'crm-bot': Number(process.env.CRM_BOT_RATE_LIMIT_PER_MIN) || 50`. Refactor moves the `50` fallback to the new `getPlatformConfig` helper. **`rate-limiter.ts` stays sync-pure** — the route handlers look up the limit via `getPlatformConfig` and call `rateLimiter.check(workspaceId, 'crm-bot', { limit: N })` where `limit` is an explicit override param. This avoids making `rateLimiter.check` async (would cascade into every tool that uses it).
- **D10:** After migration + deploy, user deletes `CRM_BOT_ENABLED`, `CRM_BOT_RATE_LIMIT_PER_MIN`, `CRM_BOT_ALERT_FROM` from Vercel dashboard (Vercel redeploy required AFTER code is live — if user deletes them first, code would read undefined during the deploy window, hence the order).

## Constraints

- Regla 3: all DB reads via `src/lib/domain/*`. Create `src/lib/domain/platform-config.ts`.
- Regla 5: migration MUST be applied in production before code that reads the table is deployed. Checkpoint.
- Regla 4: update `docs/analysis/04-estado-actual-plataforma.md` to note the config relocation.
- No breaking changes to existing Vercel env var behavior DURING transition — the new helper reads from DB first, env var as fallback, so code works in both states.

## Explicit non-goals

- Admin UI (future)
- Per-workspace overrides (future)
- Full observability on config changes (future — Phase 42.1 already captures endpoint traffic)
- Migrating OTHER env vars (only the 3 CRM bot ones; Anthropic keys, Supabase URL, etc. stay in Vercel)
- Rewriting `rate-limiter.ts` internals — keep sync API, only route callers resolve the limit
