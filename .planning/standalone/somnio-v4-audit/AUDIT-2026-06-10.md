### 2.2 Path A vs Path B (semántica de interrupción)

| | Path A (combined restart) | Path B (reprocess) |
|---|---|---|
| Condición | Interrupt con **0 templates enviados** | Interrupt con **≥1 template enviado** |
| Acción | Drena pending + `clearInterrupt` + `dropOwnEntry` + combina `[priorMsg, ...pending].join('\n')` → re-run en MISMA lambda con MISMO lock | Descarta templates restantes del turno, drena pending, hereda `carryState` (intentsVistos, templatesEnviados+sent, datosCapturados, pack, ledger dims) → re-run solo con mensajes nuevos |
| Persistencia | CERO escrituras DB durante restart (Pitfall 8) | Acumula `accumulatedSentContents` cross-iteración |
| Sitios de detección | 5: CKPT-0 (:237), agent-discriminator (:452), CKPT-6a (:549), CKPT-6b 0-sends (:669), send-loop 0-sends (:861) | 2: CKPT-6b 1+ sends (:713), send-loop 1+ sends (:885) |
| Edge | Interrupt + pending VACÍO + 0 sends → fallback legacy `_v3:pendingUserMessage` cross-lambda (:916) | Interrupt + pending vacío + 1+ sends → finish normal (:741) |

**No hay debounce previo:** el lock SET NX es el anti-debounce. Followers no disparan Inngest; el holder drena y combina todo de golpe.
