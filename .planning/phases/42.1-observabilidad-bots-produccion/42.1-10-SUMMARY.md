**Key rendering decisions:**

- `key={selectedTurn.id}` on `TurnDetailView` in index.tsx forces a clean remount whenever the user clicks a different row — cheaper than reconciling in-flight fetch state and guarantees the previous turn's data never flashes.
- EventRow owns its own expanded state (local `useState`) — expanding multiple rows at once is natural, and collapsing is cheap because the body unmounts (freeing large JsonView trees).
- `@uiw/react-json-view` theme is selected via `useTheme().resolvedTheme` (next-themes) — dark/light switch matches the rest of the inbox.
- Collapsible sub-sections in AiCallView (System Prompt / Messages / Response) default to **closed** because system prompts can exceed 10k tokens; the user opts in per row.

## TurnDetail Shape (Repository Contract)

```typescript
interface TurnDetail {
  turn: TurnSummary & { error: { name, message, stack? } | null }
  events: Array<{
    id, sequence, recordedAt, category, label, payload, durationMs
  }>
  queries: Array<{
    id, sequence, recordedAt, tableName, operation, filters,
    columns, requestBody, durationMs, statusCode, rowCount, error
  }>
  aiCalls: Array<{
    id, sequence, recordedAt, promptVersionId, purpose, model,
    messages, responseContent,
    inputTokens, outputTokens,
    cacheCreationInputTokens, cacheReadInputTokens, totalTokens,
    costUsd, durationMs, statusCode, error
  }>
  promptVersionsById: Record<string, {
    id, promptHash, systemPrompt, model,
    temperature, maxTokens, provider, firstSeenAt
  }>
}
```

**camelCase normalization:** DB columns (`prompt_hash`, `duration_ms`, `total_cost_usd`, etc.) are mapped to camelCase in the repository — the UI never touches snake_case. Numerics stored as NUMERIC (Postgrest returns strings) are coerced via `Number()` at the repo boundary.

**prompt version dereferencing:** `getTurnDetail` collects `Set(ai_calls.prompt_version_id)`, does a single `.in('id', ids)` on `agent_prompt_versions`, and builds `promptVersionsById` as a pre-joined map. This avoids an N+1 round trip from the UI and lets the AI call view render the full system prompt with O(1) lookup. If a prompt version is missing (race between a manual DB prune and the UI read), the AI call view falls back to showing the promptVersionId and an italic "no encontrada" message — non-fatal.

## Partition Pruning Strategy

`agent_observability_turns` is partitioned `BY RANGE (started_at)` with composite PK `(started_at, id)`. Filtering by `id` alone would force Postgres to scan every partition because `id` is not in the partition expression. The master pane (Plan 09) already has `startedAt` for every row in the list, so we pass it through `getTurnDetailAction(turnId, startedAt)` → `getTurnDetail(turnId, startedAt)`.

The repository then applies a +/-60s window on `started_at`:

```typescript
const turnRangeStart = new Date(turnStart.getTime() - 60_000).toISOString()
const turnRangeEnd = new Date(turnStart.getTime() + 60_000).toISOString()

supabase
  .from('agent_observability_turns')
  .select('*')
  .eq('id', turnId)
  .gte('started_at', turnRangeStart)
  .lte('started_at', turnRangeEnd)
  .single()
```

This guarantees the query touches **exactly one** monthly partition (the one containing the turn's start). The 60s buffer is purely to avoid edge cases at midnight-of-month-1 where clock skew between the UI clock and the DB's `started_at` could theoretically straddle the boundary. It is not functional: a turn either exists or it doesn't, and the `.eq('id', turnId)` predicate is the actual filter — the `started_at` bounds only prune partitions.

**Child tables (events / queries / ai_calls):** intentionally NOT filtered by `recorded_at` today. They are partitioned by `recorded_at` and indexed by `(turn_id, sequence)` which is selective enough during MVP. A post-MVP optimization is documented inline in `getTurnDetail` (see comment block around line 130): once perf becomes a concern, fetch the turn row first, then use its `finished_at` as the upper bound of a `recorded_at` window filter on the three child queries. This would prune child partitions to typically one or two instead of scanning the full month. We chose NOT to do this now because:

1. Hardcoding a 10-minute window would silently drop data for any legitimately long turn (tool calls with backoff retries can take minutes).
2. Splitting into two round trips (turn → children) adds latency that dwarfs the scan savings for today's data volume.
3. The `(turn_id, sequence)` indices on each child table already make the scan extremely selective.

The optimization is a one-line addition per child query when the time comes and the comment tells the future reader exactly how.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Data-fetching pattern replacement (SWR → hand-rolled useEffect)**

- **Found during:** Task 2 planning phase (prompt explicitly flagged this)
- **Issue:** The plan text in 10-PLAN.md showed a `useSWR` implementation, but Plan 09 had already discovered the repo does not depend on SWR (no `swr` in package.json) and deliberately hand-rolled the master pane with `useEffect + setInterval + inFlightRef` to avoid adding a new dep in Phase 42.1. Adopting `useSWR` just for the detail view would have (a) broken the build on a missing dep, (b) violated the "zero new deps" constraint of the phase, and (c) introduced two different data-fetching conventions inside the same panel directory.
- **Fix:** Mirrored the turn-list.tsx pattern for one-shot fetch — `useEffect(() => { fetchOnce() }, [turnId, startedAt])` with a local `cancelled` flag and `mountedRef` to avoid setState-after-unmount. NO polling, NO revalidation, NO interval — intentional per Pitfall 7 (turnos inmutables post-flush). The pattern is actually cleaner than SWR for this use case because there's genuinely nothing to revalidate.
- **Files modified:** `src/app/(dashboard)/whatsapp/components/debug-panel-production/turn-detail.tsx` (new file)
- **Commit:** `7d1551b`

### Automation-only Changes

**2. Added `key={selectedTurn.id}` on TurnDetailView instance in index.tsx**

Not strictly a deviation but worth flagging: the plan did not specify a React key strategy for switching turns. Adding `key=` forces remount on selection change, guaranteeing clean fetch state without having to manually track which turnId is in flight. Matches the "session switching via key-based remount" pattern already documented in MEMORY.md.

**3. Added `min-w-0 min-h-0` to detail pane wrapper in index.tsx**

Needed because the parent flex container can otherwise prevent the inner timeline from becoming shorter than its content, breaking the scrollable overflow. Standard flexbox escape hatch.

## Decisions Made

1. **No SWR** — mirrored Plan 09's hand-rolled pattern. Zero new deps in Phase 42.1 preserved.
2. **+/-60s partition pruning buffer** — purely defensive against clock skew at month boundaries, not functional filtering. The `.eq('id', turnId)` remains the actual selector.
3. **Child tables unfiltered by recorded_at** — `(turn_id, sequence)` indices are selective enough; optimization documented inline for post-MVP.
4. **Collapsible sub-sections default closed in AiCallView** — system prompts can be huge, user opts in.
5. **Theme-aware JsonView** — use `resolvedTheme` from next-themes, consistent with sandbox debug panel.
6. **`key={turnId}` on TurnDetailView** — clean remount on selection change, no in-flight fetch state to reconcile.
