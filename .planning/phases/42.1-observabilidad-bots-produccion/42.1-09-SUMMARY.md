## Super-User Detection Mechanism

**Finding:** The repo has NO pre-existing "super-user / owner of the whole platform" concept. What exists:

- `workspace_members.role` ‚Üí 'owner' | 'admin' | 'agent' scoped per-workspace.
- `usePermissions` ‚Üí role-based permission helper per-workspace.
- Grep for `SUPER_USER|isSuperUser` before this plan ‚Üí 0 matches in `src/`.

**Decision:** Introduce a new minimal env-var mechanism rather than overloading `workspace_members.role === 'owner'` (which already has meaning ‚Äî an owner of a single workspace is not the same as "Jose, owner of the whole platform"). The env-var approach:

1. Is explicit and auditable in Vercel settings.
2. Fails closed: unset env var ‚Üí `getIsSuperUser()` returns `false` ‚Üí button invisible for everyone.
3. Is trivially revocable (remove the env var ‚Üí instant kill switch).
4. Does not pollute a public database table with a concept that only exists for Jose.
5. Can be replaced later with a dedicated `platform_admins` table or a `user_metadata.is_platform_admin` flag without touching any consumer (just update `getIsSuperUser`).

**New files:** `src/lib/auth/super-user.ts`

- `SUPER_USER_EMAIL_ENV` ‚Äî constant with the env var name for safe referencing in messages / runbooks.
- `getSuperUserEmail()` ‚Äî reads + normalizes the env var.
- `getIsSuperUser()` ‚Äî async, uses `createClient()` (SSR Supabase) to resolve the authenticated user and compares lowercased email. Try/catch wrapped ‚Üí any error returns `false`.
- `assertSuperUser()` ‚Äî throws generic `Error('FORBIDDEN')` for use in server actions, zero info leakage.

**Plan 11 runbook must add:** `SUPER_USER_EMAIL=jose@morfx.app` (or whatever address Jose uses in the Supabase Auth) to Vercel Production + Preview env var scopes. Until then, the debug panel is invisible to everyone ‚Äî which is the intended fail-closed default during rollout.

## Allotment Integration Approach

`allotment@1.20.5` was already in `package.json` (added by Plan research) and is already used in two other surfaces: `sandbox/components/sandbox-split-panel.tsx` and `comandos/components/comandos-split-panel.tsx`. The CSS import pattern is `import 'allotment/dist/style.css'` ‚Äî imported locally inside `inbox-layout.tsx` so no global stylesheet changes were needed.

**Conditional rendering rule:** the Allotment wrapper is ONLY mounted when all three conditions hold:

```ts
debugPanelOpen && isSuperUser && selectedConversationId
```

If any of those is false the layout falls through to the plain `<ChatView flex-1 />` as before. This is intentional: Allotment uses absolute positioning inside the pane, so wrapping ChatView unconditionally would be a visual regression for regular users (it alters flex behavior, adds drag handles, forces min sizes, etc.). Regla 6 requires the regular-user experience to be byte-identical to the pre-plan state, and branching at the JSX level is the simplest way to guarantee that.

**ChatView height:** added `h-full` to ChatView's root `<div>` so that when it is rendered inside an `Allotment.Pane` (which gives absolute height, not flex), the internal flex column still has a height context. In the non-Allotment branch the parent `<div className="flex h-full">` already provides the same context, so there is no visual difference.

## TurnList State Machine

Hand-rolled polling (no SWR / react-query in repo ‚Äî verified via `grep` before deciding). The component exposes five mutually exclusive view states:

| State      | Trigger                                    | UI                                                                                       |
| ---------- | ------------------------------------------ | ---------------------------------------------------------------------------------------- |
| `loading`  | First fetch after conversationId change    | "Cargando turnos..." (muted)                                                             |
| `disabled` | Server returns `{ status: 'disabled' }`    | Amber text + `OBSERVABILITY_ENABLED=true` env var name in a code block                   |
| `empty`    | Server returns `{ status: 'ok', turns:[]}` | "Sin turnos registrados para esta conversacion." (muted)                                 |
| `data`     | Server returns `{ status: 'ok', turns:[‚Ä¶]}`| Scrollable button list with timestamp, duration, agent+trigger, counters, cost per turn |
| `error`    | Thrown error / network failure             | Red text + error message + "Reintentar" button (calls `fetchOnce` directly)              |

**Polling contract:**

- 15 000 ms interval, cleared on unmount or conversationId change.
- `inFlightRef` skips ticks that overlap the previous request (important because server actions can take >15s under load, and stacking requests would thrash the DB).
- `mountedRef` guards all `setView` calls, preventing setState-after-unmount warnings when the user rapidly switches conversations.
- First fetch fires immediately on mount (not after the first 15s tick).

**Discriminated union rationale:** the three conditions "flag off", "flag on / no data", and "flag on / has data" look identical to a caller that only sees `TurnSummary[]`. Returning `{ status: 'disabled' }` vs `{ status: 'ok' }` lets the UI surface a genuinely actionable message in the disabled case ("go set this env var in Vercel") without invoking the client-side flag check (which would be wrong anyway ‚Äî the client does not have access to `process.env.OBSERVABILITY_ENABLED`).

## Deviations from Plan

**None material.** Minor adjustments applied (Rule 2 / Rule 3):

1. **No SWR.** The plan tentatively suggested SWR but flagged "confirmar cual". Verified `package.json` has neither `swr` nor `@tanstack/react-query`. Used a hand-rolled `useEffect + setInterval` pattern with an in-flight ref. Adding a new data-fetching dependency for one pane would be gratuitous.
2. **No existing super-user helper.** Verified via grep ‚Äî nothing like `isSuperUser`, `SUPER_USER`, etc. existed in `src/`. Followed the plan's fallback path (env var) and extracted a reusable helper at `src/lib/auth/super-user.ts` so Plan 10 (turn detail) and Plan 11 (runbook) can reuse the same functions.
3. **Super-user forwarded as prop.** Instead of accessing `process.env` from a client component, resolved `getIsSuperUser()` in the Server Component (`whatsapp/page.tsx`) and forwarded the boolean down to `InboxLayout`. Keeps the client bundle clean and aligns with how `clientConfig` is already plumbed.
4. **TurnList renders 5 states instead of the 3 the plan mentions** (added `loading` and `error` as first-class). Plan's three were the happy-path surfaces; `loading` and `error` were implicit. Making them explicit avoids flicker and dead-ends.
5. **Conditional Allotment wrapper.** The plan suggested always rendering `<Allotment>` with a conditional `<Allotment.Pane>`. But unconditionally wrapping ChatView in an Allotment would measurably change the regular-user layout (absolute positioning inside the pane, min-size constraints, drag handle). Switched to a branch-at-the-JSX-level approach that guarantees pixel-identical output for non-super-users ‚Äî strictly safer under Regla 6.
6. **ChatView `h-full` class.** Added so the component works inside both a flex parent (current inbox) and an Allotment.Pane (absolute positioning). Zero visible effect in the flex case because the parent already has `h-full`.

Since the agent cannot take screenshots, describing the target visual states:

**State 1 ‚Äî Regular user (isSuperUser=false):**

**State 4 ‚Äî Super-user, debug panel open, flag ON, no turns yet:**
Same as State 3 but left sub-pane shows muted "Sin turnos registrados para esta conversacion.". Polls every 15s, so once the first turn flushes the list populates without manual refresh.

**State 5 ‚Äî Super-user, debug panel open, flag ON, turns present:**
Left sub-pane is a vertical list of turn cards (newest first). Each card has: timestamp (es-CO / Bogota, DD/MM HH:MM:SS), duration ms, agent id + trigger kind (ERROR badge in red if `hasError`), counters row (`3ev 8q 2ai 1420tok $0.0032`). Selected turn has `bg-muted`. Right sub-pane currently shows the turn id + started_at + "Detalle del turno ‚Äî implementado en Plan 10." (stub for Plan 10).

1. **New `SUPER_USER_EMAIL` env var, not a DB-backed role.** Rationale in "Super-User Detection Mechanism" above.

4. **Conditional Allotment wrapper (not unconditional with conditional pane).** Guarantees Regla 6 byte-identical rendering on the regular-user path.
5. **Stubbed `getTurnDetail` in the repository now.** Plan 10 can fill the body without re-exporting from a different file ‚Äî single surface for both read functions.
