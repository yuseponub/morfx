# Phase 999.1: WhatsApp interactive message composer (botones + lista) - Research

**Researched:** 2026-06-03
**Domain:** WhatsApp Cloud API interactive (session) messages + operator inbox composer UI + provider-aware domain chokepoint
**Confidence:** HIGH (internal patterns verified by reading every referenced file; external Meta constraints cross-verified across 3+ sources)

## Summary

This phase is ~80% internal pattern-replication and ~20% external research. The sending primitives already exist and are unit-tested (Phase 39): `metaWhatsappSender.sendButtons` / `.sendList` with clamps, `sendWhatsAppInteractive` envelope, and the legacy 360dialog `sendButtonMessage`. What is missing is (1) a domain chokepoint `sendInteractiveMessage` that mirrors `sendTextMessage`/`sendMediaMessage`/`sendTemplateMessage` exactly (read provider once via `readWhatsappProvider`, branch, store JSONB, return `DomainResult<SendMessageResult>`), (2) a server action mirroring `sendMessage` (auth → 24h-window check → resolve apiKey → build ctx → call domain → revalidate), (3) an operator composer entry-point in the inbox toolbar (`message-input.tsx`) gated by the open 24h window, and (4) a rich outbound bubble in `message-bubble.tsx` consuming the stored JSONB.

The two genuinely-external unknowns are now resolved. **Finding #1 (UX):** interactive buttons/lists do **NOT** exist in the native WhatsApp Business app composer — they are API-only [VERIFIED]. So the user's mental model ("a button in the textbox like WhatsApp's app") describes a BSP/inbox affordance, not a native WhatsApp one. The correct analog is the existing in-repo `TemplateButton` → modal pattern: a toolbar button opens a builder dialog with add-button/add-row controls, a live WhatsApp-style preview, inline limit validation, and a send action — gated to the open 24h window. **Finding #2 (constraints):** the existing clamps (≤3 buttons / title ≤20 / ≤10 sections / row title ≤24 / list button ≤20) are CURRENT and correct, but **incomplete** — they miss several Meta limits (body ≤1024, footer ≤60, header text ≤60, row description ≤72, **total rows across ALL sections ≤10**, button id ≤256). These gaps belong in the composer's inline validation (D-05a) even though the existing sender clamps are the backstop.

**Primary recommendation:** Replicate the Phase 39 chokepoint verbatim for a single union-discriminated `sendInteractiveMessage(ctx, params)`; build the composer as a `TemplateButton`-style toolbar button → Dialog with live preview and inline validation covering the COMPLETE Meta limit set (not just the 5 clamped dimensions); store the full interactive structure in `messages.content` JSONB so the outbound bubble renders rich. Do NOT touch the 360dialog `sendButtonMessage` function (add only its caller); lists stay meta_direct-only.

### Pattern 1: Provider chokepoint (Phase 39 — replicate EXACTLY)
**What:** A single domain function reads `whatsapp_provider` once and branches. Creds resolve from `ctx.workspaceId`. 360dialog arm stays byte-identical.
**When to use:** Always, for `sendInteractiveMessage`. This is the locked decision D-03.
**Example (model after `sendTemplateMessage`):**
```typescript
// Source: src/lib/domain/messages.ts:372-462 (sendTemplateMessage), adapted
export interface SendInteractiveMessageParams {
  conversationId: string
  contactPhone: string
  apiKey: string                       // 360dialog key — caller resolves (meta arm ignores it)
  interactiveType: 'buttons' | 'list'  // D-04 union discriminant
  body: string
  header?: string
  footer?: string
  buttons?: Array<{ id: string; title: string }>            // interactiveType==='buttons'
  buttonLabel?: string                                       // interactiveType==='list'
  sections?: Array<{ title: string; rows: Array<{ id: string; title: string; description?: string }> }>
}

export async function sendInteractiveMessage(
  ctx: DomainContext,
  params: SendInteractiveMessageParams
): Promise<DomainResult<SendMessageResult>> {
  const supabase = createAdminClient()
  try {
    const provider = await readWhatsappProvider(supabase, ctx.workspaceId)  // ◄── ONCE
    let wamid: string | undefined

    if (provider === 'meta_direct') {
      const creds = await resolveByWorkspace(ctx.workspaceId, 'whatsapp')   // T-39-02
      if (!creds?.accessToken || !creds.phoneNumberId) {
        return { success: false, error: 'Credenciales Meta no configuradas' }
      }
      const metaCreds = { accessToken: creds.accessToken, phoneNumberId: creds.phoneNumberId }
      const resp = params.interactiveType === 'buttons'
        ? await metaWhatsappSender.sendButtons(metaCreds, params.contactPhone, params.body, params.buttons ?? [], params.header, params.footer)
        : await metaWhatsappSender.sendList(metaCreds, params.contactPhone, params.body, params.buttonLabel ?? '', params.sections ?? [], params.header, params.footer)
      wamid = resp.externalMessageId
    } else {
      // 360dialog (legacy — zero change to sendButtonMessage; D-03 / Regla 6)
      if (params.interactiveType === 'list') {
        return { success: false, error: 'lista no soportada en 360dialog (legacy)' }
      }
      const resp = await sendButtonMessage(params.apiKey, params.contactPhone, params.body, params.buttons ?? [], params.header, params.footer)
      wamid = resp.messages?.[0]?.id
    }

    if (!wamid) return { success: false, error: 'No se recibio ID de mensaje de WhatsApp' }

    // Store FULL interactive structure as JSONB (D-04) so the outbound bubble renders rich.
    const content = { interactiveType: params.interactiveType, body: params.body,
      ...(params.header ? { header: params.header } : {}), ...(params.footer ? { footer: params.footer } : {}),
      ...(params.buttons ? { buttons: params.buttons } : {}),
      ...(params.buttonLabel ? { buttonLabel: params.buttonLabel } : {}),
      ...(params.sections ? { sections: params.sections } : {}) }

    const { data: message, error: insertError } = await supabase.from('messages').insert({
      conversation_id: params.conversationId, workspace_id: ctx.workspaceId, wamid,
      direction: 'outbound', type: 'interactive', content, status: 'sent',
      timestamp: new Date().toISOString(),
    }).select('id').single()
    // ... insertError guard + conversations update (preview '[Interactivo]') identical to siblings
    return { success: true, data: { messageId: message!.id, waMessageId: wamid } }
  } catch (error) { /* identical catch as siblings */ }
}
```
> NOTE: import `sendButtonMessage` into `messages.ts`. The existing imports alias 360dialog funcs (`send360Text`, etc.) at the top (L21-24) — add `sendButtonMessage as send360Buttons` to that import block.

### Anti-Patterns to Avoid
- **Deciding provider at the call-site:** The server action / UI must NEVER read `whatsapp_provider` or branch on it. Only `domain.sendInteractiveMessage` decides (Regla 3 / D-03).
- **Modifying `sendButtonMessage` (360dialog):** D-03 says byte-identical. Only add the caller. Do not "improve" its clamps.
- **Adding a 360dialog list function:** Explicitly deferred (D-03 / Deferred Ideas). 360dialog list → clear domain error, not a new function.
- **Rendering interactive via `dangerouslySetInnerHTML`:** The template-builder bubble deliberately avoids it (React escapes; `whitespace-pre-wrap` is CSS). Match that.
- **Trusting only the 5 sender clamps for UX validation:** The clamps miss several limits (see Pitfalls). The composer's inline validation must cover the FULL set.

## Common Pitfalls

### Pitfall 1: Inline validation only covers the 5 clamped dimensions (INCOMPLETE limits)
**What goes wrong:** Operator types a 1500-char body, a 90-char footer, an 11th total row, or a 100-char row description; the sender's `.slice` clamps (≤3 btn / title ≤20 / ≤10 sections / row title ≤24 / list btn ≤20) silently mangle or Meta rejects the send.
**Why it happens:** The existing clamps (meta-whatsapp-sender.ts:122-173) DON'T enforce: body ≤1024, header text ≤60, footer ≤60, **total rows across ALL sections ≤10** (a list with 10 sections × 2 rows = 20 rows → REJECTED), row description ≤72, button id ≤256, list section title ≤24.
**How to avoid:** The D-05a validation helper (`interactive-limits.ts`) must enforce the COMPLETE set below and block send (or show field errors) BEFORE calling the action. Sender clamps remain backstop only.
**Warning signs:** Meta error `#131009` (parameter value invalid) / `#100` (invalid parameter) on send; truncated titles in the customer's WhatsApp.

**COMPLETE current Meta interactive limit set (cross-verified):**

|-------|---------|------|-------------------|
| body text | ≤1024 | ≤1024 | HIGH [VERIFIED multi-source] |
| header (text) | ≤60 | ≤60 | HIGH [VERIFIED] |

### Pitfall 2: Interactive button placed in the closed-window branch
**What goes wrong:** Operator can open the composer when the 24h window is CLOSED; Meta rejects every interactive send (interactive = session/free-form message, requires open window).
**Why it happens:** `message-input.tsx` has TWO returns — closed-window (L291-333) and open-window (L335+). Adding the button to the wrong branch (or both).
**How to avoid:** Render the interactive button ONLY in the open-window return branch (D-02). The server action ALSO re-checks the window (defense-in-depth, mirror `sendMessage` L114-127). [VERIFIED: closed branch only offers `TemplateButton`.]
**Warning signs:** "Ventana de 24h cerrada" / Meta `#131047` re-engagement error on interactive send.

### Pitfall 3: 360dialog list send not blocked
**What goes wrong:** A 360dialog workspace tries to send a list; there is no 360dialog list function, so it crashes or silently no-ops.
**Why it happens:** Forgetting the `interactiveType==='list'` guard on the 360dialog arm.
**How to avoid:** D-03 explicit: 360dialog + list → `return { success:false, error:'lista no soportada en 360dialog (legacy)' }`. The composer SHOULD also hide/disable the "list" type for 360dialog workspaces if it can know the provider client-side — but since provider is a domain-only concern (Regla 3), the safer approach is: let the action attempt and surface the domain error via toast. (Most live workspaces are 360dialog today; meta_direct is only the test workspace `4b5d84dd` per STATE.md — so list will mostly hit this path in practice. Surface the error clearly.)
**Warning signs:** Operator confusion when a list "disappears" on a 360dialog workspace.

### Pitfall 4: Button/row id collisions or empty ids
**What goes wrong:** Two buttons share an id, or an id is empty → the inbound `button_reply`/`list_reply` reply is ambiguous; webhook stores the *title* as text (see below) so an empty/dup id may not break rendering but breaks any future automation keyed on id.
**Why it happens:** Auto-generating ids naively (e.g. index only) or letting them be blank.
**How to avoid (Claude's discretion D-05/id):** Auto-generate stable unique ids — recommend `btn_${slug(title)}_${index}` or a short nanoid, enforce uniqueness in validation. Simplest robust choice: `crypto.randomUUID()` slice or `${type}_${index}` guaranteed-unique within the message. Do NOT require the operator to type ids.
**Warning signs:** Duplicate-id render glitches; future automation can't disambiguate.

### Pitfall 5: Inbound interactive reply is stored as text — don't assume a rich inbound bubble exists
**What goes wrong:** Planner assumes there is an existing inbound interactive *rich* bubble to copy. There isn't.
**Why it happens:** The webhook handler (`webhook-handler.ts:806-851`) converts inbound `button_reply`/`list_reply` to **plain text** (`body: msg.interactive.button_reply?.title`), storing them as `type` text-equivalent. `message-bubble.tsx case 'interactive'` is only the OUTBOUND placeholder (L133-140). [VERIFIED]
**How to avoid:** D-05c is about the OUTBOUND bubble only (the message the operator just sent). Build the rich render from scratch (reusing the preview component), not by copying a non-existent inbound interactive bubble.
**Warning signs:** Searching for an inbound interactive renderer and finding only the text path.

### Pitfall 6: Breaking the 3 GREEN sender tests / the provider parity tests (Regla 6)
**What goes wrong:** Touching `meta-whatsapp-sender.ts` or `sendButtonMessage` breaks `meta-whatsapp-sender.test.ts` (3/3) or the 360dialog-parity assertions.
**How to avoid:** Do NOT edit those files. `sendInteractiveMessage` only *imports and calls* them. New tests go in NEW files mirroring `messages-provider.test.ts`.

### Validation helper (NEW — the only genuinely-new pure logic, D-05a)
```typescript
// src/lib/whatsapp/interactive-limits.ts  (NEW — unit-tested)
export const INTERACTIVE_LIMITS = {
  body: 1024, header: 60, footer: 60,
  maxButtons: 3, buttonTitle: 20, buttonId: 256,
  listButtonLabel: 20, maxSections: 10, sectionTitle: 24,
  maxTotalRows: 10, rowTitle: 24, rowDescription: 72, rowId: 200,
} as const

export interface InteractiveValidationError { field: string; message: string }

export function validateButtons(p: { body: string; header?: string; footer?: string;
  buttons: { id: string; title: string }[] }): InteractiveValidationError[] {
  const errs: InteractiveValidationError[] = []
  if (!p.body.trim()) errs.push({ field: 'body', message: 'El cuerpo es obligatorio' })
  if (p.body.length > INTERACTIVE_LIMITS.body) errs.push({ field: 'body', message: `Máx ${INTERACTIVE_LIMITS.body} caracteres` })
  if ((p.header?.length ?? 0) > INTERACTIVE_LIMITS.header) errs.push({ field: 'header', message: `Máx ${INTERACTIVE_LIMITS.header}` })
  if ((p.footer?.length ?? 0) > INTERACTIVE_LIMITS.footer) errs.push({ field: 'footer', message: `Máx ${INTERACTIVE_LIMITS.footer}` })
  if (p.buttons.length < 1) errs.push({ field: 'buttons', message: 'Al menos 1 botón' })
  if (p.buttons.length > INTERACTIVE_LIMITS.maxButtons) errs.push({ field: 'buttons', message: `Máx ${INTERACTIVE_LIMITS.maxButtons} botones` })
  p.buttons.forEach((b, i) => {
    if (!b.title.trim()) errs.push({ field: `button.${i}`, message: 'Título obligatorio' })
    if (b.title.length > INTERACTIVE_LIMITS.buttonTitle) errs.push({ field: `button.${i}`, message: `Máx ${INTERACTIVE_LIMITS.buttonTitle}` })
  })
  const ids = p.buttons.map(b => b.id)
  if (new Set(ids).size !== ids.length) errs.push({ field: 'buttons', message: 'IDs de botón duplicados' })
  return errs
}

export function validateList(p: { body: string; header?: string; footer?: string; buttonLabel: string;
  sections: { title: string; rows: { id: string; title: string; description?: string }[] }[] }): InteractiveValidationError[] {
  const errs: InteractiveValidationError[] = []
  if (!p.body.trim()) errs.push({ field: 'body', message: 'El cuerpo es obligatorio' })
  if (p.body.length > INTERACTIVE_LIMITS.body) errs.push({ field: 'body', message: `Máx ${INTERACTIVE_LIMITS.body}` })
  if (!p.buttonLabel.trim()) errs.push({ field: 'buttonLabel', message: 'Etiqueta del botón obligatoria' })
  if (p.buttonLabel.length > INTERACTIVE_LIMITS.listButtonLabel) errs.push({ field: 'buttonLabel', message: `Máx ${INTERACTIVE_LIMITS.listButtonLabel}` })
  if (p.sections.length > INTERACTIVE_LIMITS.maxSections) errs.push({ field: 'sections', message: `Máx ${INTERACTIVE_LIMITS.maxSections} secciones` })
  const totalRows = p.sections.reduce((n, s) => n + s.rows.length, 0)
  if (totalRows < 1) errs.push({ field: 'sections', message: 'Al menos 1 fila' })
  if (totalRows > INTERACTIVE_LIMITS.maxTotalRows) errs.push({ field: 'sections', message: `Máx ${INTERACTIVE_LIMITS.maxTotalRows} filas en total` }) // ◄ Pitfall 1
  p.sections.forEach((s, si) => {
    if (s.title.length > INTERACTIVE_LIMITS.sectionTitle) errs.push({ field: `section.${si}`, message: `Título máx ${INTERACTIVE_LIMITS.sectionTitle}` })
    s.rows.forEach((r, ri) => {
      if (!r.title.trim()) errs.push({ field: `row.${si}.${ri}`, message: 'Título obligatorio' })
      if (r.title.length > INTERACTIVE_LIMITS.rowTitle) errs.push({ field: `row.${si}.${ri}`, message: `Máx ${INTERACTIVE_LIMITS.rowTitle}` })
      if ((r.description?.length ?? 0) > INTERACTIVE_LIMITS.rowDescription) errs.push({ field: `row.${si}.${ri}`, message: `Descripción máx ${INTERACTIVE_LIMITS.rowDescription}` })
    })
  })
  const allRowIds = p.sections.flatMap(s => s.rows.map(r => r.id))
  if (new Set(allRowIds).size !== allRowIds.length) errs.push({ field: 'rows', message: 'IDs de fila duplicados' })
  return errs
}
```
