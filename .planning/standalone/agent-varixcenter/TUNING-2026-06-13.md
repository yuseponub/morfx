## BUG CRÍTICO encontrado: timers L1-L6 nunca se disparaban

**Síntoma (usuario):** "no veo que estén sirviendo L1, L2".
**Evidencia:** 40 sesiones / 183 turnos → 0 acciones de retoma (`origen:'timer'`), 0 eventos de timer en observability, 0 mensajes de retoma enviados.

**Causa raíz:** `createProductionAdapters` (`engine-adapters/production/index.ts`) solo activaba el `V3ProductionTimerAdapter` (el que tiene `emitSignals`) para `somnio-sales-v3` y `godentist`. Para `varixcenter` caía al `ProductionTimerAdapter` v1 (sin `emitSignals`) → en el runner, `if (output.timerSignals.length > 0 && 'emitSignals' in this.adapters.timer)` daba **false** → los timerSignals del agente (L1/L2/etc.) se **descartaban en silencio**. Los eventos `agent/v3.timer.started` nunca se emitían a Inngest.

El agente Varix SÍ emitía los signals correctamente (sales-track + transitions #8 L2, #2 L1, etc.); el dispatcher de timers (sitio #7, hotfix previo) también estaba listo. El eslabón roto era la **selección del adapter**.

**Fix (2 puntos):**
1. `index.ts`: `varixcenter` agregado al set `V3_TIMER_AGENTS` (allowlist de `usesV3Timer`).
2. `v3-timer.ts`: `TIMER_TABLES['varixcenter'] = { real: VARIX_TIMER_DURATIONS }` (duraciones propias L1=180s, L2=120s, L3=120s, L4=120s, L5=180s, L6=90s; sin esto caía a las de somnio).

**Verificación:** vitest 130/130, tsc=0. Live tras deploy.

**⚠️ `godentist-fb-ig` tiene EL MISMO bug** (no está en el allowlist → sus timers tampoco emiten; el hotfix previo solo arregló el lado RECEIVE/dispatcher, no el EMIT). NO lo toqué en este commit (Regla 6 — agente en prod con leads reales). Pendiente de confirmación del usuario para aplicar el mismo fix.

**Nota menor:** el bloque de timers ENCADENADOS en `agent-timers-v3.ts` usa `V3_TIMER_DURATIONS` (somnio) para todos los agentes. Varix casi no encadena (1 retoma por fase), así que el impacto es despreciable; queda como deuda menor.
