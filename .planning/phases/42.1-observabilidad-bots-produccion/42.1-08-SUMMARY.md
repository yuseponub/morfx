# Phase 42.1 Plan 08: Observability Partition Purge Cron Summary

**One-liner:** Cron Inngest diario 03:00 Bogota que crea particion mes-siguiente e idempotentemente dropea particiones agent_observability_* mas viejas que 30 dias via RPCs del schema.

## Objetivo Cumplido

Automatizar el lifecycle de las 4 tablas particionadas de observabilidad (`agent_observability_turns`, `events`, `queries`, `ai_calls`) para garantizar:

1. **Retencion de 30 dias** sin intervencion manual (Decision #3 del CONTEXT)
2. **Creacion proactiva** de la particion del mes siguiente, evitando que el dia 1 de cada mes falle por ausencia de destino de INSERT
3. **Patron DROP PARTITION vs DELETE** (del RESEARCH.md): O(1) metadata-only vs O(N) row-by-row

## Schedule

| Cron string | Timezone | Hora local | Contexto |
|---|---|---|---|
| `0 3 * * *` | `TZ=America/Bogota` | 03:00 | 1 hora despues de `close-stale-sessions` (02:00) para evitar overlap de presion sobre DB en ventana low-traffic |

- `retries: 1` (idempotente: si la ejecucion previa fallo, la siguiente simplemente re-crea/re-ignora y re-dropea)
- Inngest v3.51.0 requiere prefijo inline `TZ=` en el cron string (no hay opcion separada)

## Steps

### Step 1: `ensure-next-month-partition`

```ts
const next = new Date()
next.setMonth(next.getMonth() + 1)
next.setDate(1)
const targetMonth = next.toISOString().slice(0, 10) // 'YYYY-MM-01'
await supabase.rpc('create_observability_partition', { target_month: targetMonth })
```

- Calcula primer dia del mes siguiente
- Llama al RPC `create_observability_partition(target_month DATE)` creado en Plan 01
- El RPC crea las 4 particiones del mes indicado (turns/events/queries/ai_calls) en una sola transaccion
- **Idempotente**: si las particiones ya existen, no-op silencioso
- Errores se re-lanzan para que Inngest reintente una vez
- Logger: `logger.info({ targetMonth }, 'ensured next-month partition exists')`

### Step 2: `drop-old-partitions`

```ts
const cutoff = new Date()
cutoff.setDate(cutoff.getDate() - 30)
cutoff.setDate(1) // floor to start of that month
const cutoffIso = cutoff.toISOString().slice(0, 10)
const { data } = await supabase.rpc('drop_observability_partitions_older_than', { cutoff: cutoffIso })
```

- `cutoff = hoy - 30 dias`, luego floored al inicio del mes
- El "flooring a inicio de mes" evita drop prematuro: solo eliminamos particiones cuyo mes completo esta mas viejo que 30 dias
- El RPC `drop_observability_partitions_older_than(cutoff DATE)` itera las particiones con sufijo `YYYYMM` menor al cutoff y las DROPea
- Retorna lista de nombres de particiones dropeadas
- Steady state: en un mes tipico se dropean 4 particiones (1 por tabla) del mes 2-atras

## Feature Flag OFF Safety

Este cron es **seguro de activar inmediatamente** incluso antes de que cualquier bot escriba observabilidad real. Razones:

1. **Opera sobre schema directo via RPC**, no depende del collector AsyncLocalStorage ni de `runWithCollector`.
2. **No activa el collector**: corre en el runtime de Inngest scheduled, fuera de cualquier turn context, asi que `createAdminClient()` (instrumentado en otras capas) no tiene collector activo donde recursar. Cero pitfall de Pitfall 1 (self-recursion).
3. **Idempotente con dataset vacio**:
   - Si no hay datos (flag OFF, nadie escribe), las particiones del mes actual tienen 0 rows pero existen; Step 1 las crea para el mes siguiente (no-op si ya existen del seed inicial).
   - Step 2 no encuentra particiones viejas que dropear (array vacio retornado). Cero rows afectadas.
4. **Sin efectos colaterales en codigo de produccion**: el cron no toca `agent_sessions`, `messages`, `contacts`, ni ninguna tabla usada por agentes/automations/webhooks.

Conclusion: deployable en el mismo push que los plans de wiring, SIN requerir activacion escalonada.

## Decisiones Hechas

1. **03:00 Bogota (no 02:00)**: 1 hora de offset despues de close-stale-sessions para evitar overlap de locks/presion sobre Postgres en la ventana nocturna. close-stale corre DELETE/UPDATE sobre `agent_sessions`, purge corre DROP PARTITION sobre tablas de observabilidad — tecnicamente no colisionan, pero el offset da margen de retry sin cruzarse.
2. **Cutoff a inicio de mes**: en lugar de `hoy - 30 dias` exacto, hacemos floor al primer dia de ese mes. Esto evita un race donde al dia 29 del mes siguiente la particion "mes anterior" aun tiene rows a los que un query legitimo podria querer acceder. Con el floor, solo dropeamos particiones cuyo mes completo ya no cabe en la ventana de 30 dias.
3. **retries: 1**: mismo que close-stale-sessions. El cron es idempotente, un segundo intento tras fallo transitorio (connection reset, deadlock) es suficiente. Mas retries solo acumulan ruido en Inngest dashboard.
