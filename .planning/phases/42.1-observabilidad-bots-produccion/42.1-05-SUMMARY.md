# Phase 42.1 Plan 05: Somnio V3 Pipeline Instrumentation Summary

**One-liner:** Wave 3 wires the production observability collector into the Somnio V3 + shared call sites: 7 Anthropic constructions migrated to `createInstrumentedAnthropic` with `runWithPurpose` labels, the WhatsApp Inngest handler structurally wrapped in `runWithCollector`, 14 hot-path events recorded in agent-production.ts, and 19 deeper events recorded across the 5 mechanism files (silence_timer, retake, ofi_inter, interruption_handling, pending_pool) — all behind null-safe collector checks so feature flag OFF is byte-identical to baseline.

### Task 2a — Handler structural wrap

`src/inngest/functions/agent-production.ts`:

- Added imports from `@/lib/observability` (`isObservabilityEnabled`, `ObservabilityCollector`, `runWithCollector`, `type AgentId`).
- Added `resolveAgentIdForWorkspace(workspaceId)` helper inline at module scope. Reads `workspace_agent_config.conversational_agent_id` via the existing `getWorkspaceAgentConfig` helper and maps the string id to the narrow `AgentId` union. Mapping table:

| `conversational_agent_id` value | Mapped `AgentId` |
|---|---|
| `'somnio-sales-v3'` | `'somnio-v3'` |
| `'godentist'` | `'godentist'` |
| `'somnio-recompra'` / `'somnio-recompra-v1'` | `'somnio-recompra'` |
| anything else (incl. `'somnio-sales-v1'`, `'somnio-sales-v2'`, missing config, read error) | `'somnio-v2'` |

Defensive try/catch wraps the lookup so the helper never throws -- REGLA 6.

- Wrapped the entire handler body in an inner `run` arrow function. The body is otherwise unchanged.
- Top of handler: `collector = isObservabilityEnabled() ? new ObservabilityCollector({...}) : null`.
- Bottom of handler: `if (collector) return await runWithCollector(collector, run); return await run()`. The wrap path also has a try/catch that calls `collector.recordError(...)` before re-throwing.
- A `finally` block contains a placeholder for Plan 07's `step.run('observability-flush', () => collector.flush())` -- documented inline. flush() is a no-op today.

### Decision recorded — `resolveAgentIdForWorkspace` lives inline in agent-production.ts

It is a single 14-line helper used by exactly one call site. Extracting it to its own file would add an import for no benefit. If a second handler (e.g. `agent-timers-v3.ts`) needs it later we will hoist it then. Documented here so the decision is not lost.

## Deviations from Plan

### Auto-fixed issues (Rules 1/2/3)

**1. [Rule 1 - Bug] `recordEvent` label parameter typed as `string | undefined`, not `string | null`**
- **Found during:** Task 2b TypeScript compilation
- **Issue:** Plan suggested `recordEvent('mode_transition', null, {...})` but the collector signature is `(category, label?: string, ...)`. `null` is not assignable.
- **Fix:** Pass `undefined` instead of `null` for label arguments where no label fits.
- **Files modified:** `src/inngest/functions/agent-production.ts` (2 occurrences via replace_all)

**2. [Rule 1 - Bug] `gateResult.reason` does not exist on every union variant**
- **Found during:** Task 2b TypeScript compilation
- **Issue:** `gateResult` is a discriminated union; the `passthrough` and `ignore` variants have no `reason` field. Plan code referenced it unconditionally.
- **Fix:** Use `'reason' in gateResult ? gateResult.reason : null` for the gate_decision payload, and dropped the `reason` field entirely from the `ignored` variant payload.
- **Files modified:** `src/inngest/functions/agent-production.ts`

### Adapt-to-real-code (no inventing)

**3. [Rule 4 borderline -> adapted instead] agent-production.ts is a thin orchestrator, not the pipeline**

- **Found during:** Task 2b
- **Plan asked for:** 12+ recordEvent calls covering classifier (text branch), intent, mode_transition, template_selection, no_repetition, block_composition, pre_send_check, char_delay, etc.
- **Reality:** agent-production.ts only has visibility into media-gate routing and the final `result` from `processMessageWithAgent`. The deep pipeline (text classifier, intent detection, template selection, no_repetition, block composition, pre-send check, char delay) lives inside `webhook-processor.ts` and the Somnio V3 per-track files.
- **Decision:** Inject 14 events covering everything that IS visible at this layer (8 categories, see Task 2b table), and DEFER the missing 4 categories to a follow-up plan that instruments `webhook-processor.ts` + the V3 pipeline body. This is per the deviation policy "If a mechanism file is structured completely differently than plan describes, ADAPT the instrumentation to the real code rather than changing the code structure."
- **Gap log:** see "Gap log" section below.

## Decisions Made

1. **Inline `resolveAgentIdForWorkspace`** rather than a new file. Single call site, 14 lines, no benefit to extracting. Will hoist if a second handler needs it later.
2. **Streaming Anthropic call in claude-client.ts left untouched.** It returns a stream object synchronously, not a promise, which makes wrapping it in `runWithPurpose` awkward. The fetch wrapper already records a coarse `ai_call_streaming` event for `text/event-stream` responses, so observability of the call existing is preserved even without a purpose label.
3. **Adapt to real code over inventing:** when the plan's expected pipeline categories did not exist in agent-production.ts, the missing 4 (`template_selection`, `no_repetition` events, `pre_send_check`, `char_delay`) were logged as gaps for follow-up plans rather than fabricated calls in the wrong file.
4. **`finally` block in the wrap is a placeholder**, not a flush call. Plan 07 will replace it with `await step.run('observability-flush', () => collector.flush())` once flush() is implemented. Today flush() is a no-op so the call is intentionally omitted.
5. **`ofi_inter` routes mapped to observable code**: route 1 = primary `ask_ofi_inter`, route 2 = secondary `ask_ofi_inter` via `mencionaInter`, route 3 = `confirmar_cambio_ofi_inter`. If CONTEXT.md's conceptual taxonomy differs, the mapping can be revisited in a UI/refinement plan without code changes.

## REGLA 6 compliance (production agent safety)

- Feature flag `OBSERVABILITY_ENABLED` OFF (default in production today): `isObservabilityEnabled()` returns false -> `collector` is null -> the handler invokes `await run()` directly with no wrap and no ALS frame -> every `collector?.recordEvent(...)` is a single null check -> every `getCollector()?.recordEvent(...)` is also a null check (ALS empty) -> behaviour is byte-identical to the pre-Plan-05 baseline.
- Anthropic migration: `createInstrumentedAnthropic` constructs the same `new Anthropic({...})` it always did, just with `makeObservableFetch(fetch, 'anthropic')` injected as the SDK fetch. When the flag is OFF the wrapper is a fast-path that calls fetch directly with zero parsing.
- `runWithPurpose` is just `purposeAls.run(purpose, fn)`. ALS overhead is negligible (single map set/get) and the purpose value is only consumed by the fetch wrapper, which is also fast-path no-op when the flag is OFF.
- `resolveAgentIdForWorkspace` only runs when the flag is ON (it is inside the ternary).
- All builds (Next 16 production) green after every commit. No regressions in TS errors beyond pre-existing test typing issues.
