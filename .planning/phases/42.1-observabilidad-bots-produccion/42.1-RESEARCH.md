# Phase 42.1: Observabilidad y Mirroring para Bots en Produccion — Research

**Researched:** 2026-04-07
**Domain:** Deep agent observability for LLM conversational bots in Next.js 16 + Vercel serverless + Supabase postgres
**Confidence:** HIGH (key architectural claims verified) / MEDIUM (volume estimates, latency numbers)

---

## Summary

The phase builds a production-only deep observability system for three conversational bots (Somnio V3, GoDentist, Recompra), persisted in the same Supabase instance, with a split panel embedded in the WhatsApp inbox. The 8 large architectural decisions are already locked in 42.1-CONTEXT.md — this research resolves the HOW.

Three findings drive the entire implementation plan:

1. **Both the Supabase client and the Anthropic SDK natively accept a `fetch` override in their constructors.** This gives us one uniform, non-invasive interception point for *all* SQL queries and *all* Claude calls without touching any domain / agent code. No Proxy objects, no monkey-patching, no Anthropic SDK introspection. This is the single most important finding of the research.

2. **Node.js `AsyncLocalStorage` is the right propagation mechanism**, works in Vercel Node runtime (the agent pipeline runs as Inngest functions, which are plain Node serverless, not Edge), has been battle-tested since Node 16, and Next.js itself uses it internally for request context. There is *no* viable alternative that doesn't require threading the collector through ~15 files of pipeline signature.

3. **Postgres declarative RANGE partitioning by month is natively supported on Supabase** with no extension needed, and dropping an expired partition is O(1) via `DROP TABLE partition_name` — matching the monthly-purge retention requirement exactly. `pg_partman` exists but adds complexity we don't need for a 3-table observability schema.

**Primary recommendation:** Build a `src/lib/observability/` module with (a) a shared `AsyncLocalStorage<ObservabilityContext>` singleton, (b) a `makeObservableFetch(originalFetch, kind)` factory used by both `createAdminClient()` and every `new Anthropic({ fetch })` call site, (c) a 4-table monthly-partitioned schema, and (d) a feature flag `OBSERVABILITY_ENABLED` that makes the whole thing a no-op in ~0ms when disabled. Plug the collector open/flush in `whatsapp-agent-processor` Inngest function (the single entry point for production turns).

### Pattern 1: AsyncLocalStorage Context Propagation (CRITICAL)

**What:** A module-level `AsyncLocalStorage` singleton holds the per-turn `ObservabilityCollector`. Any code downstream in the same async call tree calls `getCollector()` and records events without having the collector passed as a parameter.

**When to use:** Always, for every event recorded by this system. This is the core mechanism that makes capture transparent.

**Key facts verified:**
- AsyncLocalStorage is available in Vercel's Node runtime (the Inngest handler runs in Node, not Edge).
- It survives `await`, `Promise.all`, `setTimeout`, and any callback scheduled on the same tick by the Node event loop (Node `async_hooks` contract).
- It does NOT survive being serialized into an Inngest `step.run()` that is retried in a *different* invocation (because that's a fresh Node process). This is fine for our use case: **one turn = one Inngest handler execution = one AsyncLocalStorage context**. If a `step.run` retries, the outer handler retries, a new context is spawned, and a new turn row is produced. This matches the "one row per turn attempt" semantic we want anyway.
- Context isolation between concurrent Inngest invocations is guaranteed by Node (each invocation has its own call tree).

**Example:**

```typescript
// src/lib/observability/context.ts
import { AsyncLocalStorage } from 'node:async_hooks'
import type { ObservabilityCollector } from './collector'

const als = new AsyncLocalStorage<ObservabilityCollector>()

export function runWithCollector<T>(
  collector: ObservabilityCollector,
  fn: () => Promise<T>,
): Promise<T> {
  return als.run(collector, fn)
}

/** Returns the active collector or null if observability is disabled / no turn active. */
export function getCollector(): ObservabilityCollector | null {
  return als.getStore() ?? null
}
```

```typescript
// src/inngest/functions/agent-production.ts — WRAP the turn body
import { runWithCollector, ObservabilityCollector } from '@/lib/observability'
import { isObservabilityEnabled } from '@/lib/observability/flag'

// ... inside the Inngest function, after media-gate, before step 3:
const collector = isObservabilityEnabled()
  ? new ObservabilityCollector({ conversationId, workspaceId, agentId: 'somnio-v3', turnStartedAt: new Date() })
  : null

const result = collector
  ? await runWithCollector(collector, () => processTurn(...))
  : await processTurn(...)

if (collector) {
  await step.run('observability-flush', () => collector.flush())
}
```

### Pattern 2: Custom `fetch` Interception (CRITICAL)

**What:** Both `createClient` (supabase-js) and `new Anthropic()` accept a `fetch` option. We provide a wrapper that (a) calls the original fetch, (b) reads the request URL/method/body, (c) times it, (d) reads the response, (e) appends an event to the current collector via `getCollector()`.

**Verified (HIGH confidence):**
- supabase-js: `createClient(url, key, { global: { fetch: customFetch } })` — documented API.
- Anthropic SDK: `new Anthropic({ apiKey, fetch: customFetch })` — documented API.

**Key insight:** Intercepting at `fetch` is STRICTLY BETTER than intercepting at the query builder level because:
- Postgrest chains (`.from('x').select('a,b').eq('id', 1)`) are lazy — they only commit to an HTTP request at `await`. At the `.from()` level you don't have filters yet.
- At the `fetch` level you get the final URL `/rest/v1/x?select=a,b&id=eq.1`, the method, the headers (including `x-workspace-id` if we add it), the body (for INSERT/UPDATE), the status, and the count.
- Parsing Postgrest URLs is straightforward and stable (it's just a REST convention).

**Example:**

```typescript
// src/lib/observability/fetch-wrapper.ts
import { getCollector } from './context'

type FetchKind = 'supabase' | 'anthropic'

export function makeObservableFetch(
  originalFetch: typeof fetch = fetch,
  kind: FetchKind,
): typeof fetch {
  return async (input, init) => {
    const collector = getCollector()
    if (!collector) return originalFetch(input, init)  // no-op fast path

    const startedAt = performance.now()
    const url = typeof input === 'string' ? input : (input instanceof URL ? input.toString() : input.url)
    const method = init?.method ?? 'GET'

    try {
      const response = await originalFetch(input, init)
      const durationMs = performance.now() - startedAt

      // Clone for body inspection — response.clone() is cheap
      if (kind === 'supabase') {
        collector.recordQuery(parsePostgrestUrl(url, method, init?.body), durationMs, response.status)
      } else {
        // Read response clone for token counts (Anthropic returns JSON)
        const cloned = response.clone()
        const body = await cloned.json().catch(() => null)
        collector.recordAiCall({ url, method, requestBody: init?.body, responseBody: body, durationMs })
      }
      return response
    } catch (err) {
      collector.recordError({ kind, url, method, error: err, durationMs: performance.now() - startedAt })
      throw err
    }
  }
}
```

### Pattern 3: Single-Batch Flush at End of Turn

**What:** The collector accumulates events in memory throughout the turn. At the very end — after the agent has produced its response and state is saved — a single `INSERT ... VALUES ...` batch persists the turn row + all events in one round-trip per table.

**Why it works here:**
- We are already inside an Inngest handler (`whatsapp-agent-processor`). The Inngest handler is not blocking the user — the webhook already responded 200 to Meta. So `await collector.flush()` adds ~15-50ms to the Inngest handler duration but ZERO user-facing latency.
- Inngest already bills by step, so wrapping the flush in `step.run('observability-flush', ...)` gives us retry-for-free and a clean separation from the agent work.
- We do NOT need Vercel's `after()` or `waitUntil()` here — those are for cases where you need the HTTP response to flush before background work, which is only relevant for the `/api/webhooks/whatsapp/route.ts` entry, NOT for the Inngest handler which is already "background".

**Example:**

```typescript
// src/lib/observability/flush.ts
export class ObservabilityCollector {
  async flush() {
    if (this.events.length === 0 && this.queries.length === 0 && this.aiCalls.length === 0) return
    const supabase = createRawAdminClient()  // IMPORTANT: use non-instrumented client to avoid recursion
    const turnId = crypto.randomUUID()

    // 1. Resolve/create prompt version rows first (each AI call FK's into this)
    const promptVersionIds = await this.resolvePromptVersions(supabase)

    // 2. Single turn row
    await supabase.from('agent_observability_turns').insert({
      id: turnId,
      conversation_id: this.conversationId,
      workspace_id: this.workspaceId,
      agent_id: this.agentId,
      started_at: this.turnStartedAt,
      finished_at: new Date(),
      event_count: this.events.length,
      query_count: this.queries.length,
      ai_call_count: this.aiCalls.length,
      total_tokens: this.totalTokens,
      total_cost_usd: this.estimateCost(),
      error: this.error ?? null,
    })

    // 3. Batch insert children (all 3 in parallel — independent tables)
    await Promise.all([
      this.events.length > 0 && supabase.from('agent_observability_events').insert(
        this.events.map(e => ({ ...e, turn_id: turnId })),
      ),
      this.queries.length > 0 && supabase.from('agent_observability_queries').insert(
        this.queries.map(q => ({ ...q, turn_id: turnId })),
      ),
      this.aiCalls.length > 0 && supabase.from('agent_observability_ai_calls').insert(
        this.aiCalls.map(a => ({ ...a, turn_id: turnId, prompt_version_id: promptVersionIds.get(a.promptHash)! })),
      ),
    ])
  }
}
```

**CRITICAL anti-recursion note:** the flush itself uses supabase-js to INSERT. If we use the instrumented client, the INSERT will recursively try to record itself into the collector, causing either an infinite loop or polluting the very events table it's trying to write. Solution: export a separate `createRawAdminClient()` (not instrumented) from `src/lib/supabase/admin.ts` for observability's internal use ONLY.

### Pattern 4: Feature Flag Kill-Switch

**What:** An env var `OBSERVABILITY_ENABLED=true|false` gates the whole system. When off, `getCollector()` returns null, the fetch wrapper falls through to the raw fetch, no collector is constructed, no AsyncLocalStorage context is created.

**Why:** Regla 6 of the project — any new system touching production agents must be instantly disableable without a redeploy being strictly required to recover. Vercel env var changes propagate on next cold start (or can be promoted immediately via `vercel env ... && vercel redeploy`).

**Overhead when disabled:** One boolean check (`process.env.OBSERVABILITY_ENABLED === 'true'`) per turn entry point + one `getCollector() === null` check per fetch call. We can cache the env var read at module load. Total cost: ~microseconds, effectively zero.

```typescript
// src/lib/observability/flag.ts
const ENABLED = process.env.OBSERVABILITY_ENABLED === 'true'
export function isObservabilityEnabled(): boolean { return ENABLED }
```

### Pattern 5: Prompt Versioning via Content Hash + `INSERT ... ON CONFLICT`

**What:** Every AI call's system prompt is hashed (SHA-256 of normalized string), and the `agent_prompt_versions` table has a `UNIQUE (prompt_hash)` constraint. Before recording an AI call event, we do an atomic upsert that returns the existing row on conflict. The AI call event row stores `prompt_version_id` (FK) instead of the full prompt text — typically 5-20K tokens saved per call.

**Race-safety in serverless:** Multiple concurrent turns can hit the upsert for a brand-new prompt at the same moment. `INSERT ... ON CONFLICT (prompt_hash) DO UPDATE SET prompt_hash = EXCLUDED.prompt_hash RETURNING id` is atomic at the Postgres level — even under concurrency, only one row is created and both callers get the same `id` back. (The `DO UPDATE` is a no-op trick to force `RETURNING` on conflict; `DO NOTHING` does not return anything on conflict.)

```sql
-- Upsert prompt version returning existing OR new id
INSERT INTO agent_prompt_versions (prompt_hash, system_prompt, model, temperature, max_tokens, provider, first_seen_at)
VALUES ($1, $2, $3, $4, $5, $6, now())
ON CONFLICT (prompt_hash) DO UPDATE SET prompt_hash = agent_prompt_versions.prompt_hash
RETURNING id
```

```typescript
// src/lib/observability/prompt-version.ts
import { createHash } from 'node:crypto'

export function hashPrompt(systemPrompt: string, model: string, params: { temperature?: number; max_tokens?: number }): string {
  // Normalize: strip leading/trailing whitespace, collapse inner whitespace, hash
  const normalized = JSON.stringify({
    p: systemPrompt.trim().replace(/\s+/g, ' '),
    m: model,
    t: params.temperature ?? null,
    x: params.max_tokens ?? null,
  })
  return createHash('sha256').update(normalized).digest('hex')
}
```

### Anti-Patterns to Avoid

- **Using a Proxy around PostgrestQueryBuilder** — the chain is lazy; you'll capture empty filter objects. Intercept at `fetch` instead.
- **Storing full prompt text on every AI call event** — Decision #7 in CONTEXT.md already resolved this, but the anti-pattern is tempting ("just in case the hash collision…"). SHA-256 collisions do not happen in this lifetime.
- **Using `Realtime` to push events to the panel** — Decision C already said no streaming. Don't accidentally introduce Realtime subscriptions "because it's cool" — it reintroduces connection count limits and complicates the kill-switch.
- **`JSON.stringify(error)` to capture errors** — loses stack trace. Capture `{ name, message, stack, cause }` explicitly.
- **Awaiting `collector.record*()` calls from the pipeline** — record methods MUST be synchronous (push to array). Only `flush()` is async.
- **Global singleton collector** — every turn gets its OWN collector instance. AsyncLocalStorage isolates them. A global singleton would leak state between turns.
- **Logging observability internals via the same observability system** — always use the existing `pino` `createModuleLogger('observability')` for internal errors. Never recurse.
- **Depending on sandbox `DebugTurn` types directly** — Decision A says paralelo. Duplicate the type shapes in `src/lib/observability/types.ts`. If you import `DebugTurn` from sandbox you couple the two systems.
- **Capturing `history` (full message history) on every turn row** — it's already persisted in `whatsapp_messages`. Instead, capture `messageId` references; the UI joins at render time.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-request context propagation | Threading `collector` parameter through every function | `node:async_hooks` `AsyncLocalStorage` | Native since Node 16, zero deps, used by Next.js internally. Any hand-rolled alternative (ContextManager, thread-local pattern) is reinventing this |
| Supabase query capture | Proxy wrapper around `PostgrestQueryBuilder`, monkey-patching the chain | `createClient(url, key, { global: { fetch: customFetch } })` | Documented, stable API; sees the final composed query |
| Anthropic call capture | Monkey-patching `Anthropic.prototype.messages.create` | `new Anthropic({ fetch: customFetch })` | Documented API; survives SDK upgrades |
| Periodic purge of old rows | `DELETE FROM ... WHERE created_at < ...` in a cron | Postgres declarative RANGE partitioning by month + `DROP TABLE partition_202601` | DELETE on hot tables causes bloat + VACUUM pressure; DROP is O(1) |
| Monthly partition creation | Hardcoded partition DDL per month | Small PL/pgSQL helper function `create_observability_partition(year, month)` called by the same Inngest cron that purges old ones | Keeps DDL out of migration files for future months |
| Cron scheduling | Custom setInterval, Vercel Cron | Inngest `createFunction` with `{ cron: 'TZ=America/Bogota ...' }` | Already the repo standard (see `close-stale-sessions.ts`) |
| UI split panel | Hand-built flex with resize handle | `allotment` — already in package.json | Drag-to-resize, persists size in localStorage out of the box |
| JSON tree rendering in panel | Custom recursive component | `@uiw/react-json-view` — already in package.json | Expand/collapse, copy-to-clipboard, search built in |
| SHA-256 prompt hash | Custom hash function | `node:crypto` `createHash('sha256')` | Native |
| URL parsing for Postgrest | Regex over query strings | `new URL(url).searchParams` | Native, handles encoding correctly |
| Timestamp zone handling | Manual UTC math | `timezone('America/Bogota', now())` in SQL | Project standard (Regla 2) |
| Token / cost estimation | Hand-written per-model table frozen at dev time | Small `src/lib/observability/pricing.ts` with versioned prices, reviewed quarterly | Anthropic publishes prices; codify the table in one file so it's findable and updatable. (No canonical npm package for Anthropic pricing exists) |

**Key insight:** The custom `fetch` option on both clients is the single most load-bearing finding of this research. Without it, we would be looking at weeks of Proxy / monkey-patch work with fragile results. With it, this phase becomes mostly a schema + UI exercise.

## Schema Recommendations

**Design note:** Normalized 4-table schema (not JSONB-consolidated). Reasons:
- Queries by filter (all queries to `conversations` table across all turns; all AI calls using prompt version X) are trivial with normalized rows, painful with JSONB.
- Inserts are still just 4 batch inserts per turn — not a performance concern.
- Partitioning works per-table transparently with normalized schema.
- JSONB for *within-event* payload (e.g. captured query filters, captured prompt params) where the structure varies — yes. JSONB for the *whole turn* — no.

```sql
-- =====================================================================
-- Migration: 20260408000000_observability_schema.sql
-- Phase 42.1: Agent observability (production bots)
-- =====================================================================

-- ---------------------------------------------------------------------
-- Table 1: agent_prompt_versions (NOT partitioned — deduplicated, small)
-- ---------------------------------------------------------------------
CREATE TABLE agent_prompt_versions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prompt_hash TEXT NOT NULL UNIQUE,             -- sha256 hex
  system_prompt TEXT NOT NULL,                  -- full text, no truncation (Decision #8)
  model TEXT NOT NULL,                          -- e.g. 'claude-haiku-4-5-20251001'
  temperature DOUBLE PRECISION,
  max_tokens INTEGER,
  provider TEXT NOT NULL DEFAULT 'anthropic',
  first_seen_at TIMESTAMPTZ NOT NULL DEFAULT timezone('America/Bogota', now()),
  last_seen_at TIMESTAMPTZ NOT NULL DEFAULT timezone('America/Bogota', now())
);

CREATE INDEX idx_agent_prompt_versions_last_seen ON agent_prompt_versions (last_seen_at DESC);

-- ---------------------------------------------------------------------
-- Table 2: agent_observability_turns (partitioned monthly by started_at)
-- ---------------------------------------------------------------------
CREATE TABLE agent_observability_turns (
  id UUID NOT NULL DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL,
  workspace_id UUID NOT NULL,
  agent_id TEXT NOT NULL,                       -- 'somnio-v3' | 'godentist' | 'somnio-recompra'
  turn_number INTEGER,                          -- optional, if agent tracks it
  started_at TIMESTAMPTZ NOT NULL,
  finished_at TIMESTAMPTZ,
  duration_ms INTEGER GENERATED ALWAYS AS (
    EXTRACT(EPOCH FROM (finished_at - started_at)) * 1000
  )::INTEGER STORED,
  event_count INTEGER NOT NULL DEFAULT 0,
  query_count INTEGER NOT NULL DEFAULT 0,
  ai_call_count INTEGER NOT NULL DEFAULT 0,
  total_tokens INTEGER NOT NULL DEFAULT 0,
  total_cost_usd NUMERIC(10, 6) NOT NULL DEFAULT 0,
  error JSONB,                                  -- { name, message, stack } or null
  trigger_message_id UUID,                      -- FK-ish to whatsapp_messages
  trigger_kind TEXT,                            -- 'user_message' | 'timer' | 'system_event'
  current_mode TEXT,                            -- for agent state snapshot
  new_mode TEXT,
  PRIMARY KEY (started_at, id)                  -- composite so partition column is in PK
) PARTITION BY RANGE (started_at);

CREATE INDEX idx_turns_conversation ON agent_observability_turns (conversation_id, started_at DESC);
CREATE INDEX idx_turns_workspace_agent ON agent_observability_turns (workspace_id, agent_id, started_at DESC);

-- ---------------------------------------------------------------------
-- Table 3: agent_observability_events (partitioned monthly by recorded_at)
-- Generic event timeline — classifier, intent, mode transitions, etc.
-- ---------------------------------------------------------------------
CREATE TABLE agent_observability_events (
  id UUID NOT NULL DEFAULT gen_random_uuid(),
  turn_id UUID NOT NULL,
  recorded_at TIMESTAMPTZ NOT NULL,
  sequence INTEGER NOT NULL,                    -- order within turn, 0..N
  category TEXT NOT NULL,                       -- 'classifier' | 'intent' | 'mode_transition' | 'template_selection' | 'no_repetition' | 'guard' | 'block_composition' | 'pre_send_check' | 'timer_signal' | 'handoff' | 'tool_call' | 'error' | ...
  label TEXT,                                   -- short human label
  payload JSONB NOT NULL,                       -- category-specific
  duration_ms INTEGER,                          -- nullable (only for timed events)

-- ---------------------------------------------------------------------
-- Partition management helper function
-- ---------------------------------------------------------------------
CREATE OR REPLACE FUNCTION create_observability_partition(target_month DATE)
RETURNS void
LANGUAGE plpgsql
AS $$
DECLARE
  suffix TEXT := to_char(target_month, 'YYYYMM');
  start_date DATE := date_trunc('month', target_month);
  end_date DATE := start_date + INTERVAL '1 month';
BEGIN
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS agent_observability_turns_%s PARTITION OF agent_observability_turns FOR VALUES FROM (%L) TO (%L)',
    suffix, start_date, end_date
  );
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS agent_observability_events_%s PARTITION OF agent_observability_events FOR VALUES FROM (%L) TO (%L)',
    suffix, start_date, end_date
  );
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS agent_observability_queries_%s PARTITION OF agent_observability_queries FOR VALUES FROM (%L) TO (%L)',
    suffix, start_date, end_date
  );
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS agent_observability_ai_calls_%s PARTITION OF agent_observability_ai_calls FOR VALUES FROM (%L) TO (%L)',
    suffix, start_date, end_date
  );
END;
$$;

CREATE OR REPLACE FUNCTION drop_observability_partitions_older_than(cutoff DATE)
RETURNS TABLE(dropped_table TEXT) LANGUAGE plpgsql AS $$
DECLARE r RECORD;
BEGIN
  FOR r IN
    SELECT inhrelid::regclass AS tbl
    FROM pg_inherits
    WHERE inhparent::regclass::text IN (
      'agent_observability_turns',
      'agent_observability_events',
      'agent_observability_queries',
      'agent_observability_ai_calls'
    )
  LOOP
    -- Parse YYYYMM from table name; drop if older than cutoff
    IF substring(r.tbl::text from '(\d{6})$')::text <
       to_char(cutoff, 'YYYYMM') THEN
      EXECUTE format('DROP TABLE %s', r.tbl);
      dropped_table := r.tbl::text;
      RETURN NEXT;
    END IF;
  END LOOP;
END;
$$;
```

## Common Pitfalls

### Pitfall 1: Infinite Recursion via Self-Instrumented Flush

**What goes wrong:** The observability flush calls `supabase.from('agent_observability_turns').insert(...)`. If the supabase client used is the instrumented one, the fetch wrapper tries to `recordQuery` onto the collector. The collector is in the middle of flushing, pushing more events, producing more queries, never terminating.

**Why it happens:** Transparent wrapping is *too* transparent — the module doesn't know its own writes are special.

**How to avoid:**
- Export TWO functions from `src/lib/supabase/admin.ts`: `createAdminClient()` (instrumented, default) and `createRawAdminClient()` (raw, no wrapper). The observability module uses `createRawAdminClient()` exclusively.
- Alternatively, the fetch wrapper can check a URL allowlist and skip tables starting with `agent_observability_`. Less robust; URL matching is fragile.
- **Recommended:** separate factory function. It's explicit and impossible to misuse.

**Warning signs:** Dev stacktraces with 500+ frames of `makeObservableFetch → record → flush → fetch → ...`. Supabase connection pool exhaustion under any turn. Log floods about the observability tables themselves.

### Pitfall 2: AsyncLocalStorage Context Lost Across Inngest `step.run` Retries

**What goes wrong:** A `step.run('orchestrator', ...)` fails, Inngest retries it. The retry may execute in a NEW invocation (fresh Node process). The original AsyncLocalStorage context is gone. The collector inside that retry is `null`. Events are silently dropped for the retried step.

**Why it happens:** Inngest's durability model — steps are idempotent and can execute on fresh workers.

**How to avoid:**
- Initialize the collector at the TOP of the Inngest handler, outside `step.run`. `runWithCollector(collector, () => step.run(...))` wraps the step so the context is in scope for the first attempt.
- Accept that step retries get a fresh collector per retry — design the flush as the LAST step so it sees only the events of the final successful attempt.
- Document this limitation: "turn-level retries produce a new turn row; step-level retries within one turn may lose partial events if steps fail in the middle". Acceptable for V1.

**Warning signs:** Turn rows with `finished_at` set but suspiciously few events/queries compared to similar turns. Events for the last part of a turn missing.

### Pitfall 3: PostgREST URL Parsing — Encoded Filter Values

**What goes wrong:** The fetch wrapper parses `url.searchParams` to capture filters. PostgREST operators look like `?id=eq.1&created_at=gte.2026-01-01`. If a filter value contains `&` or `=` (e.g. a search by URL), URL encoding can make parsing non-trivial.

**How to avoid:**
- Use native `new URL(url).searchParams` — it handles decoding correctly.
- Store filters as `Record<string, string>` after parsing, never re-parse the raw query string downstream.
- For inserts/updates (where filters aren't in URL but body is in `init.body`), record the body as JSONB without trying to interpret it.

**Warning signs:** `filters` column containing garbled or truncated values; dashboard showing wrong WHERE clauses.

### Pitfall 4: Prompt Hash Collision with `JSON.stringify` Key Order

**What goes wrong:** `JSON.stringify({ a: 1, b: 2 })` and `JSON.stringify({ b: 2, a: 1 })` produce different strings → different hashes → two prompt version rows for the same logical prompt.

**How to avoid:** When normalizing, use an explicit, fixed-order serialization (see `hashPrompt` example above with keys `p`, `m`, `t`, `x`). Do NOT `JSON.stringify(params)` directly.

**Warning signs:** `agent_prompt_versions` table growing unreasonably fast; duplicate rows with slightly different `created_at` but identical content.

### Pitfall 5: Feature Flag Evaluated Once at Module Load

**What goes wrong:** `const ENABLED = process.env.OBSERVABILITY_ENABLED === 'true'` reads the env var at module load time. In Vercel serverless, this is per cold start. If you flip the flag and don't cause a new cold start, old function instances keep the old value for minutes.

**How to avoid:**
- For Vercel, this is usually acceptable — env var changes trigger a new deployment or can be promoted via `vercel env` + redeploy.
- If you want zero-downtime toggle, read `process.env.OBSERVABILITY_ENABLED` on every call. Cost is negligible.
- **Recommended:** evaluate on every call. The cost is one property access. Simpler mental model.

**Warning signs:** "I turned it off but logs keep coming" — likely warm Vercel functions still have the old value.

### Pitfall 6: Batch INSERT Too Large for a Single Request

**What goes wrong:** A pathological turn produces 200+ events (e.g. heavy no-repetition filter on a long pending pool) and the batch INSERT payload becomes >1MB, hitting PostgREST's default body size limit or triggering slow parsing.

**How to avoid:**
- Set a soft cap in the collector: if `events.length > 100`, the collector should chunk the insert into batches of 100. Same for queries and AI calls.
- Log a warning (pino) when a turn exceeds the soft cap so we can investigate if a bot is misbehaving.
- The Postgres partitioning design means we don't need to worry about index bloat from the chunking — each batch still hits the same partition.

**Warning signs:** Rare errors "request body too large"; turn rows with `event_count: 200+`.

### Pitfall 7: Panel UI Auto-Refresh Storm

**What goes wrong:** The panel auto-refreshes every N seconds. If the user leaves the panel open on a busy conversation with 50 turns, every refresh re-queries all 50 turn detail rows with all their events. Multiply by 5 open tabs. Supabase connection count climbs.

**How to avoid:**
- Use SWR or React Query with a reasonable `refreshInterval` (10-15s), and `dedupingInterval` ≥ 5s.
- The list view (turn-list) should refresh. The turn-detail should NOT auto-refresh — once a turn is flushed, it's immutable. Fetch on click and cache forever.
- Use `stale-while-revalidate` semantics.

**Warning signs:** "Supabase connection count warning"; panel flickering.

### Pitfall 8: Coupling to Sandbox `DebugTurn` Types

**What goes wrong:** Natural temptation to `import type { DebugTurn } from '@/lib/sandbox/types'`. Then a future sandbox refactor breaks production observability, violating Decision A.

**How to avoid:** Define `ProdObservabilityTurn` shape in `src/lib/observability/types.ts`. Even if fields overlap 80% with `DebugTurn` today, they are separate types. Add an ESLint rule if possible: `no-restricted-imports` with pattern `@/lib/sandbox/types` from `@/lib/observability/**`.

**Warning signs:** Search shows `import ... from '@/lib/sandbox` inside `src/lib/observability/`.

### Pitfall 9: Forgetting to Instrument One of the 10 Anthropic Call Sites

**What goes wrong:** The codebase has 10 distinct `new Anthropic({ ... })` instantiations. Miss one → silent gap in AI call capture.

**How to avoid:**
- Create ONE helper: `src/lib/observability/anthropic-instrumented.ts`

  ```typescript
  export function createInstrumentedAnthropic(opts: { apiKey?: string } = {}) {
    return new Anthropic({
      apiKey: opts.apiKey ?? process.env.ANTHROPIC_API_KEY,
      fetch: makeObservableFetch(fetch, 'anthropic'),
    })
  }
  ```
- Migrate all 10 call sites to use it. Then add an ESLint rule or grep-gate in CI: `grep -rn "new Anthropic(" src/lib/agents | wc -l` should return 0 (except inside the helper itself).
- **File inventory of call sites (verified):**
  - `src/lib/agents/claude-client.ts:46`
  - `src/lib/agents/media/sticker-interpreter.ts:54`
  - `src/lib/agents/somnio/minifrase-generator.ts:67`
  - `src/lib/agents/somnio/message-classifier.ts:130`
  - `src/lib/agents/somnio/template-paraphraser.ts:45`
  - `src/lib/agents/somnio/no-repetition-filter.ts:78`
  - `src/lib/agents/somnio-recompra/comprehension.ts:30`
  - `src/lib/agents/godentist/comprehension.ts:27`
  - `src/lib/agents/somnio-v2/comprehension.ts:27` (NOTE: v2, may not be in scope of Phase 42.1 bots but instrument anyway for safety)

**Warning signs:** AI call counts on turn rows don't match expected per-agent totals; `agent_observability_ai_calls` missing rows for specific `purpose` categories.

### Pitfall 10: Postgres Partition Primary Key Gotcha

**What goes wrong:** Forgetting that the partition key column must be part of the primary key. Writing `PRIMARY KEY (id)` on a table partitioned by `started_at` will fail at `CREATE TABLE`.

**How to avoid:** Always use composite PK `(partition_column, id)`. This is reflected in the schema above. Foreign keys INTO a partitioned table also have caveats — we use logical UUIDs without hard FK enforcement across partitioned children (a pragmatic choice; the module writes all rows in one transaction so referential integrity is maintained at the app layer).

**Warning signs:** Migration fails with "unique constraint ... must include all partitioning columns".

### Example: Parse PostgREST URL in fetch wrapper

```typescript
// src/lib/observability/fetch-wrapper.ts — helper
export function parsePostgrestUrl(
  url: string,
  method: string,
  body: BodyInit | null | undefined,
): {
  table_name: string
  operation: 'select' | 'insert' | 'update' | 'delete' | 'rpc' | 'unknown'
  filters: Record<string, string>
  columns: string | null
  request_body: unknown
} {
  const u = new URL(url)
  // PostgREST paths: /rest/v1/<table> or /rest/v1/rpc/<fn>
  const match = u.pathname.match(/\/rest\/v1\/(rpc\/)?([^/?]+)/)
  const isRpc = !!match?.[1]
  const table_name = match?.[2] ?? 'unknown'

  const filters: Record<string, string> = {}
  let columns: string | null = null
  u.searchParams.forEach((v, k) => {
    if (k === 'select') columns = v
    else filters[k] = v
  })

  const operation =
    isRpc ? 'rpc' :
    method === 'GET' ? 'select' :
    method === 'POST' ? 'insert' :
    method === 'PATCH' ? 'update' :
    method === 'DELETE' ? 'delete' :
    'unknown'

  let request_body: unknown = null
  if (body && typeof body === 'string') {
    try { request_body = JSON.parse(body) } catch { request_body = body }
  }

  return { table_name, operation, filters, columns, request_body }
}
```

### Example: Daily purge cron (mirror of close-stale-sessions)

```typescript
// src/inngest/functions/observability-purge.ts
import { inngest } from '../client'
import { createAdminClient } from '@/lib/supabase/admin'  // using raw admin is fine here — we need writes
import { createModuleLogger } from '@/lib/audit/logger'

const logger = createModuleLogger('observability-purge')

export const observabilityPurgeCron = inngest.createFunction(
  { id: 'observability-purge', name: 'Observability Partition Purge', retries: 1 },
  { cron: 'TZ=America/Bogota 0 3 * * *' },  // 03:00 Bogota, after close-stale-sessions (02:00)
  async ({ step }) => {
    // Step 1: Create next month's partition if it doesn't exist yet
    await step.run('ensure-next-month-partition', async () => {
      const supabase = createAdminClient()
      const next = new Date(); next.setMonth(next.getMonth() + 1); next.setDate(1)
      const { error } = await supabase.rpc('create_observability_partition', { target_month: next.toISOString().slice(0, 10) })
      if (error) throw error
    })

    // Step 2: Drop partitions older than 30 days (i.e. month < 30 days ago)
    const dropped = await step.run('drop-old-partitions', async () => {
      const supabase = createAdminClient()
      const cutoff = new Date(); cutoff.setDate(cutoff.getDate() - 30); cutoff.setDate(1)
      const { data, error } = await supabase.rpc('drop_observability_partitions_older_than', { cutoff: cutoff.toISOString().slice(0, 10) })
      if (error) throw error
      return data
    })

    logger.info({ droppedPartitions: dropped, runAt: new Date().toISOString() }, 'observability purge complete')
    return { droppedPartitions: dropped }
  }
)
```
