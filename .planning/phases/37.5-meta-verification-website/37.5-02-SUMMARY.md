## Decisions Made

1. **Stub Header/Footer in Task 1 commit, flesh out in Tasks 2/3** â€” Layout imports `Header` and `Footer` from Task 2/3 files. To keep tsc clean per-task atomic commits, Task 1 creates stub versions that render empty `<header className="border-b" />` and `<footer className="border-t" />`. Tasks 2/3 overwrite them. Rationale: atomic commits > non-atomic single commit; stubs add zero runtime value but preserve the "each commit compiles" invariant.

2. **Server Header + server Footer + client LocaleToggle** â€” Only the locale switcher needs interactivity. Everything else (logo, nav links, CTAs, footer) stays server for minimal JS bundle. `getTranslations('Header'|'Footer')` server-side.

3. **LocaleLink (next-intl) only for locale-aware routes** â€” `/privacy`, `/terms`, product anchors use `Link` from `@/i18n/navigation`. `/login` uses plain `next/link` because auth pages are outside the `(marketing)/[locale]/` route group and would 404 if the locale-prefix wrapper tried to rewrite them.

4. **Bottom legal bar NOT translated** â€” Legal entity name, NIT, address, CIIU code are identical strings in both languages (legally required as-is). Hardcoded in footer.tsx rather than duplicated across es.json/en.json. Reduces translation surface area and prevents accidental mistranslation of legal identifiers.

5. **`useTransition` around `router.replace` in LocaleToggle** â€” Prevents UI jank during locale swap; button disables while pending.

6. **Accept root `<html lang="es">` from `src/app/layout.tsx`** â€” Plan explicitly authorized deferring dynamic `lang` attribute to avoid complicating root layout during fast-ship phase. Minor SEO cost on /en pages; Meta reviewers don't care; revisit if search rankings matter.
