## Flujo del Flush

```
flushCollector(collector)
  |
  +-- 0. Empty turn fast-path (debug log + return)
  |
  +-- 1. createRawAdminClient()         <-- CRITICAL: NO instrumentado
  |
  +-- 2. randomUUID() -> turnId
  |
  +-- 3. Dedup prompt versions
  |     |
  |     +-- Build Map<promptHash, PromptVersionInput> from collector.aiCalls
  |     +-- resolvePromptVersions(supabase, map) -> Map<hash, prompt_version_id>
  |
  +-- 4. INSERT 1 row in agent_observability_turns
  |     (id=turnId, conversation/workspace/agent ids, started/finished_at,
  |      counts, total_tokens, total_cost_usd, error JSONB, trigger meta, modes)
  |
  +-- 5. Promise.all([...]) — 3 paralelos:
  |     |
  |     +-- insertChunked('agent_observability_events', eventRows, 100)
  |     +-- insertChunked('agent_observability_queries', queryRows, 100)
  |     +-- insertChunked('agent_observability_ai_calls', aiCallRows, 100)
  |
  +-- 6. logger.info(timing + counts) <- p50/p95 trackeable
  |
  +-- 7. Soft-cap warnings (counts >100 or duration >200ms)
  |
  +-- catch: logger.error(...) + return  <-- REGLA 6: SWALLOW
```

**Total round-trips por turno tipico:** 5 (1 prompts upsert + 1 turn insert + 3 children insertes paralelos). Para turnos chicos esto es ~5 viajes serializados desde la perspectiva del wall clock = ~3 round-trips secuenciales (prompts -> turn -> children-paralelos).

## Decision: Swallow-on-error

**Pregunta:** ¿Que pasa si Postgres rechaza el batch insert (ej: outage transitorio, particion futura sin crear, indice corrupto)?

**Opciones consideradas:**

| Opcion | Pro | Con | Elegida |
|---|---|---|---|
| Re-throw -> Inngest retry | Garantia eventual de persistencia | El step de flush es el ULTIMO; un retry corre TODO el handler de nuevo (incluyendo Anthropic calls + sends WhatsApp duplicados). Cliente real ve mensajes repetidos | NO |
| Swallow + log error completo | Cero impacto sobre el cliente productivo. Datos perdidos quedan en logs forensicos | Datos del turno fallido se pierden | **SI** |
| Cola de retry separada | Ambos | Complejidad extra (DLQ, dedup, etc.) — Plan futuro si lo necesita | NO (aplazado) |

**Decision:** Swallow. **Razon:** La REGLA 6 del proyecto es absoluta — el bot productivo NO puede degradarse por observabilidad. Un mensaje duplicado al cliente es worse-case real (perdida de venta, confusion); un turno perdido en la tabla de observabilidad es worst-case interno (un debug menos rico). Trade off claro a favor del cliente.

**Impacto operacional:** El operador monitorea el log line `'observability flush failed — events dropped'` con counts. Una correlacion entre turnos exitosos en `agent_sessions` y filas faltantes en `agent_observability_turns` permite detectar drops sin bloquear nada.

## Decision: Flush en finally (try/catch wrapping run)

**Pregunta:** ¿Donde corre el flush si `run()` throws?

**Decision:** El flush SIEMPRE corre, antes del re-throw del error. Justificacion: los turnos fallidos son los mas interesantes de inspeccionar — perdiendolos en error path serias contraproducente. La secuencia exacta es:

1. `try { turnResult = await runWithCollector(collector, run) }`
2. `catch (err) { collector.recordError(...); turnError = err }`
3. `await step.run('observability-flush', () => collector.flush())`
4. `if (turnError) throw turnError` — re-throw para que Inngest retries y concurrency limits funcionen normalmente

El `recordError` corre **antes** del flush para que el `error` JSONB del turn row lleve la causa real del fallo.

## Timing budget (esperado)

| Operacion | p50 esperado | p95 esperado |
|---|---|---|
| Empty turn fast path | <1ms | <2ms |
| Dedup prompts (1-3 prompts) | 5-15ms | 30ms |
| Insert turn row | 10-20ms | 40ms |
| 3 children insertes paralelos (~30 events, ~10 queries, ~3 ai_calls) | 15-30ms | 80ms |
| **Total flush** | **30-65ms** | **<200ms** |

**Soft cap warning:** `durationMs > 200` dispara `logger.warn`. Cualquier outlier indica:
- Particion del mes futuro no creada (Plan 08 hara cron preventivo)
- Lock contention en Postgres
- Payload anormalmente grande (turn con cientos de eventos -> bug en pipeline)

**Hard cap:** ninguno. Si la red de Supabase muere, el flush se queda colgado hasta el timeout natural del fetch (~30s) y entonces swallowea. El step.run de Inngest lo absorbe y sigue.

## Soft caps y chunking

```
CHUNK_SIZE = 100  // Postgrest payload-friendly
```

- Turnos > 100 events / queries / ai_calls disparan warning + chunked insertes seriales por tabla
- Las 3 tablas siguen corriendo en paralelo (Promise.all) entre si — solo el chunking interno es secuencial dentro de una misma tabla
- Trade off: una particion cae a la mitad del segundo chunk -> el primer chunk queda persistido. Atomicidad NO garantizada cross-chunk. Documentado en flush.ts; aceptable porque (a) el caso es rarisimo, (b) el turn row ya esta persistido y permite re-detectar drops via row counts vs counts en columnas

## Compatibilidad con feature flag OFF (REGLA 6)

```typescript
if (!collector) {
  return await run()
}
// ...rest only when collector is non-null
```

Path corto identico al baseline pre-42.1: cero step.run extra, cero await extras, cero allocation. Verificado por trace mental + tsc clean. La unica diferencia versus pre-42.1 es la rama `if (!collector)` que el optimizador de v8 deja a cero costo.

   - **Files modified:** `src/lib/observability/flush.ts`

2. **[Rule 1 — Bug] `queries.columns` schema mismatch (TEXT vs string[])**
   - **Found during:** Task 1 mapping del insert de queries
   - **Issue:** `ObservabilityQuery.columns` es `string[] | null`. La columna del schema es `TEXT` (no `text[]`). Insertar el array directo genera `[object Object]` o un string raro al pasar por Postgrest.
   - **Fix:** `q.columns === null ? null : JSON.stringify(q.columns)`. La UI puede `JSON.parse` de vuelta cuando lea.
   - **Files modified:** `src/lib/observability/flush.ts`
   - **Commit:** `3631d44`

3. **[Rule 1 — Bug] `prompt_version_id` puede ser undefined si dedup falla**
   - **Found during:** Task 1
   - **Issue:** El template hacia `promptVersionIds.get(a.promptHash)!` con non-null assertion. Si el upsert falla en silencio (improbable, pero posible) eso seria un crash a runtime.
   - **Fix:** `promptVersionIds.get(a.promptHash) ?? null`. La columna es NOT NULL en schema, asi que el insert fallaria con un error claro en Postgres -> swallow -> log -> drop turn (en vez de throw inesperado).
   - **Commit:** `3631d44`

4. **[Rule 2 — Missing critical] No se cubria el error path en wiring del handler**
   - **Found during:** Task 2 review del codigo de Plan 05
   - **Issue:** El placeholder de Plan 05 tenia un `try/catch/finally` con flush comentado, pero el `finally` no podia ejecutarlo porque era el unico lugar donde podia correr. La logica re-thrownia el error sin que el flush hubiera persistido el `recordError`.
   - **Fix:** Reestructurar a `try { runWithCollector } catch { recordError + capture err } / step.run flush / if err throw err`. Esto garantiza el orden critico recordError -> flush -> rethrow.
   - **Commit:** `02919aa`
