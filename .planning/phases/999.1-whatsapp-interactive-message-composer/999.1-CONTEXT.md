# Phase 999.1: WhatsApp interactive message composer (botones + lista) - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Construir el **envío** de mensajes interactivos de WhatsApp (reply buttons ≤3 + lista con secciones ≤10), que hoy NO existe end-to-end:

1. **UI de operador** para componer y ENVIAR un mensaje interactivo desde el inbox.
2. **Domain `sendInteractiveMessage`** provider-aware en `src/lib/domain/messages.ts`, siguiendo el MISMO chokepoint único de la Fase 39 (lee `whatsapp_provider` UNA vez; `meta_direct` → `metaWhatsappSender`; `360dialog` → función legacy existente). NUNCA decidir provider por call-site (Regla 3).
3. **Tests** (unit + shape del payload + límites).

**Ya existe (verificado, NO redescubrir):**
- Primitiva de envío Meta: `sendWhatsAppInteractive` (`src/lib/meta/api.ts`) + `metaWhatsappSender.sendButtons` / `.sendList` (`src/lib/channels/meta-whatsapp-sender.ts`) con clamping (≤3 botones / título ≤20 / ≤10 secciones / row title ≤24 / button label ≤20). Unit-tested 3/3 GREEN en `meta-whatsapp-sender.test.ts`.
- `sendButtonMessage` 360dialog (`src/lib/whatsapp/api.ts:192`) — existe pero SIN caller (legacy). 360dialog **NO** tiene función de lista.
- Render ENTRANTE de respuestas (`button_reply` / `list_reply`) ya existe en `message-bubble.tsx`.

**NO existe hoy:** ningún domain `sendInteractiveMessage` ni UI de composición. Eso es lo nuevo.

**Fuera de alcance (otras fases / backlog):** biblioteca de plantillas interactivas reutilizables guardadas en `/configuracion/whatsapp`; disparo de interactivos desde automatizaciones o desde un agente AI.

</domain>

<decisions>
## Implementation Decisions

### UI surface (dónde se compone y envía)
- **D-01:** El composer vive en el **inbox**, NO en `/configuracion/whatsapp`. Se implementa como un **botón nuevo en la toolbar de `message-input.tsx`** (junto a attach / emoji / `TemplateButton`) que abre un Popover/Dialog para armar el mensaje interactivo y enviarlo a la conversación abierta. *(Esta decisión SUPERSEDE la elección inicial "/configuracion/whatsapp": el usuario refinó su intención describiendo el patrón de las apps de WhatsApp — "un botón en el textbox" — que compone los interactivos desde dentro del chat. La respuesta refinada manda.)*
- **D-02:** El botón de composición **solo se activa cuando el mensaje interactivo realmente puede enviarse** — es decir, cuando la **ventana de 24h está abierta** (mismo gating que ya deshabilita el input de texto cuando la ventana está cerrada). Los interactivos son *session messages*, no templates: sin ventana abierta Meta los rechaza. Cuando la ventana esté cerrada → botón oculto/deshabilitado.
- **Investigar (researcher):** patrón UX real de la WhatsApp Business app para componer botones/lista desde el chat (entry point en la toolbar, layout del builder, preview). Ver `<specifics>`.

### Provider-aware domain (chokepoint Fase 39)
- **D-03:** `sendInteractiveMessage` lee `whatsapp_provider` **UNA sola vez** (helper `readWhatsappProvider`, ya existe) y hace branch (Regla 3, idéntico a `sendTextMessage`/`sendMediaMessage`/`sendTemplateMessage`):
  - `meta_direct` → `metaWhatsappSender.sendButtons` / `.sendList` (capacidad completa botones + lista; creds vía `resolveByWorkspace(ctx.workspaceId, 'whatsapp')` — NUNCA del input).
  - `360dialog` (**legacy — "dejar atrás" = cero inversión nueva, pero NO romper, Regla 6**):
    - botones → reusa la función existente `sendButtonMessage` de `whatsapp/api.ts` **byte-idéntica** (esta fase solo le agrega el caller que hoy no tiene; cero cambios a la función).
    - lista → **NO soportada en 360dialog**; el domain retorna un error claro (p. ej. `error: 'lista no soportada en 360dialog (legacy)'`). Las listas son **meta_direct-only**.
- **D-04:** Una sola función `sendInteractiveMessage(ctx, params)` con **union discriminada** `interactiveType: 'buttons' | 'list'` (NO dos funciones separadas — el roadmap pide singular). Guarda en `messages.content` (JSONB, `type='interactive'`) la **estructura interactiva completa** enviada (body/header/footer + buttons o sections/rows) para que el bubble saliente pueda renderizar rico. Devuelve `DomainResult<SendMessageResult>` igual que las otras tres funciones de envío.

### UX del composer (alcance completo — delegado a Claude)
- **D-05a:** **Validación inline de límites** en el form: ≤3 botones (título ≤20), ≤10 secciones (row title ≤24, list button label ≤20). El clamp del sender (`.slice`) queda como backstop, pero el operador ve/respeta el límite ANTES de enviar.
- **D-05b:** **Preview estilo WhatsApp** del mensaje (burbuja con botones/lista) antes de enviar, reusando el patrón visual de `whatsapp-bubble.tsx` del template builder.
- **D-05c:** **Render rico del bubble saliente**: mejorar `message-bubble.tsx` `case 'interactive'` (hoy muestra placeholder "Mensaje interactivo — Ver en WhatsApp") para renderizar los botones/lista efectivamente enviados, consumiendo el `content` JSONB de D-04. *(El render ENTRANTE de respuestas ya existe; esto es el render del mensaje SALIENTE que el operador acaba de mandar.)*
- **D-05d:** Respetar ventana 24h (= D-02).

### Tests
- **D-06:** Mirror del patrón `src/lib/domain/__tests__/messages-provider.test.ts`:
  - shape del payload interactivo (buttons + list) para `meta_direct`.
  - branch de provider (`meta_direct` vs `360dialog`) — `whatsapp_provider` leído una vez.
  - 360dialog: botones → `sendButtonMessage`; lista → error claro (D-03).
  - validación de límites (helper del composer) — unit.
  - Mantener verdes los 3/3 de `meta-whatsapp-sender.test.ts` (Regla 6).

### Claude's Discretion
- Popover vs Dialog para el builder (lo que mejor encaje con el patrón existente de `message-input.tsx`).
- Generación de `id` de cada botón/row (auto-slug vs manual) — elegir lo más simple y robusto.
- Si hace falta un server action nuevo (`sendInteractiveMessage` en `app/actions/messages.ts`) que resuelve `apiKey`/creds y llama al domain — seguir el patrón exacto de `sendMessage`/`sendMediaMessage`.
- Necesidad (o no) de un toggle a nivel workspace para "activar" el feature: por defecto NO se requiere flag porque (a) es un envío manual iniciado por operador, no un path de agente en producción (Regla 6 no aplica al comportamiento del agente), y (b) el gating natural es la ventana 24h (D-02). Si research recomienda un flag, planificarlo.

</decisions>

## Existing Code Insights

### Reusable Assets
- `readWhatsappProvider(supabase, workspaceId)` — helper de decisión de provider, ya existe; reutilizar tal cual.
- `metaWhatsappSender.sendButtons` / `.sendList` — primitivas listas con clamping; el domain solo las invoca.
- `sendButtonMessage` (360dialog) — función legacy lista; el domain le agrega el caller para botones (cero cambios a la función → Regla 6).
- `resolveByWorkspace(ctx.workspaceId, 'whatsapp')` — resuelve creds Meta desde el workspace (T-39-02: nunca del input).
- `whatsapp-bubble.tsx` (template builder) — base visual para el preview.
- Patrón `Popover` + toolbar de `message-input.tsx` — base para el entry point del composer.

### Established Patterns
- Chokepoint único de provider (Fase 39 / Regla 3): la decisión de provider vive SOLO en el domain, leída una vez por workspace. Call-sites nunca deciden provider.
- Server action → resuelve apiKey/creds → arma `DomainContext` → llama domain → `revalidatePath`.
- `messages.content` JSONB por `type`; outbound NO emite triggers.
- Gating de ventana 24h en el input (disabled state) — los session messages requieren ventana abierta.

### Integration Points
- `message-input.tsx`: nuevo botón en la toolbar → abre composer (D-01).
- `messages.ts`: nueva `sendInteractiveMessage` después de `sendTemplateMessage` (D-03/D-04).
- `app/actions/messages.ts`: posible nuevo server action `sendInteractiveMessage` (discreción).
- `message-bubble.tsx`: enriquecer `case 'interactive'` (D-05c).

</code_context>

<specifics>
## Specific Ideas

- El usuario imagina el patrón de **WhatsApp Business app**: un botón dentro del área de composición (textbox) — "si agregan algo al textbox como un botón, eso" — que abre el armador de botones/lista. "Preferiblemente que solo se active cuando se tengan estos mensajes activos" → interpretado como: el botón solo aparece/habilita cuando el interactivo es enviable (ventana 24h abierta).
- El usuario delegó explícitamente las decisiones técnicas ("no sé nada de esto, elige tú, lo más completo") → se tomó el alcance completo en UX (D-05) y la forma de domain más completa (D-04).
- "360dialog es legacy, lo vamos a dejar atrás" → interpretado como cero inversión nueva en 360dialog (reusar la función de botones existente; listas meta-only), SIN romper el path 360dialog (Regla 6). **Confirmar en plan-review si esta interpretación es correcta.**

</specifics>
