Implement the pre-send check in the ProductionMessagingAdapter send loop. Before each template (including the first), query the DB for any new inbound message with timestamp greater than the trigger message. If found, stop the sequence. Also update the MessagingAdapter interface to return interruption info, and wire messageTimestamp through the Inngest function to the processor.

Purpose: This is the core interruption detection mechanism. Without it, the bot keeps sending templates even when the customer has already replied with a new question.
Output: Updated messaging adapter with per-template DB check, updated interface, Inngest function passing timestamp.

  **1b. Implement hasNewInboundMessage in ProductionMessagingAdapter:**

  Add a private method:
  ```typescript
  /**
   * Check if a new inbound message arrived after the trigger message.
   * Uses the existing idx_messages_conversation index on (conversation_id, timestamp DESC).
   * This is a lightweight count query (head: true) — no row data fetched.
   */
  private async hasNewInboundMessage(
    conversationId: string,
    afterTimestamp: string
  ): Promise<boolean> {
    const supabase = createAdminClient()
    const { count } = await supabase
      .from('messages')
      .select('*', { count: 'exact', head: true })
      .eq('conversation_id', conversationId)
      .eq('direction', 'inbound')
      .gt('timestamp', afterTimestamp)
    return (count ?? 0) > 0
  }
  ```

  **1c. Modify the send loop:**

  The current loop is:
  ```
  for each template:
    1. Apply char delay (if responseSpeed > 0)
    2. Send via domain
  ```

  Change to:
  ```
  for each template:
    1. Apply char delay (if responseSpeed > 0)
    2. PRE-SEND CHECK: if triggerTimestamp provided, call hasNewInboundMessage
    3. If new message found: break with interrupted=true
    4. Send via domain
    5. sentCount++
  ```

  Key implementation details:
  - The pre-send check runs AFTER the delay (so the customer has time to type during the delay) and BEFORE the actual send.
  - Applies to EVERY template including i=0 (the first one). CONTEXT.md is explicit about this.
  - If `triggerTimestamp` is not provided (e.g., sandbox or timer-triggered calls), skip the check entirely.
  - Track `interruptedAtIndex` as the value of `i` when interruption is detected.
  - Return `{ messagesSent: sentCount, interrupted: true, interruptedAtIndex: i }` on interruption.
  - On normal completion: return `{ messagesSent: sentCount }` (no interrupted field, or `interrupted: false`).

  **CRITICAL:** Do NOT save pending templates in this adapter. The adapter only detects interruption and returns the info. The caller (UnifiedEngine in Plan 04) handles pending storage based on the result.

  Log the pre-send check result at debug level:
  ```typescript

  ```
