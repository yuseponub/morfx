# Phase 38 — Plan 06: Wire register into onboarding (Plan 04 gap-closure)

## Why this plan exists

Plan 04 SPECIFIED that `meta-onboarding.ts` call `registerPhoneNumber` after subscribe
("calls it only if subscribed_apps indicates the number is unregistered" — 38-04-PLAN.md:133).
The executor created the helper in `embedded-signup.ts` but **omitted the call site**.
Consequence proven live (2026-06-03): an Embedded-Signup-connected number stays
`status: PENDING / platform_type: NOT_APPLICABLE` and **receives nothing** until someone
runs `/register` manually. So deliverable 2 (self-serve connect) does NOT leave the number
usable. This closes that deviation and handles the full activation chain documented in
`PLAYBOOK-number-activation.md`.

## Goal

After a successful Embedded Signup, the onboarding action ACTIVATES the number on Cloud API
(register), or — if blocked by a known chain step (leftover 2SV, missing payment method) —
persists an actionable `registration_status` and returns a clear Spanish message instead of
leaving a silent dead number.

## Pre-req (Regla 5 — user applies BEFORE code push)

Migration `supabase/migrations/20260603140000_meta_account_registration_status.sql`:
adds `registration_status` (enum-checked, default `pending`), `registration_error`,
`two_step_pin_encrypted`. Backfills the smoke number `1134593926408063` → `connected`.

## Tasks

### T1 — Domain: registration updater + pin on the row (Regla 3)
- `src/lib/domain/meta-accounts.ts`:
  - Extend `UpsertMetaAccountParams` with optional `twoStepPinEncrypted?: string | null`
    (persisted on insert/update when provided; left untouched when omitted).
  - Add `updateMetaAccountRegistration({ workspaceId, phoneNumberId, status, error?, twoStepPinEncrypted? })`
    — admin client, scoped by `workspace_id` + `phone_number_id` (Regla 3), updates
    `registration_status` (+ `registration_error`, + `two_step_pin_encrypted` on success).
    Returns `DomainResult<{ id }>`; never throws; never logs the pin.
- NO new getter here (connected-state read UI stays the deferred Plan 05 follow-up).

### T2 — Onboarding: call register with full chain handling
- `src/app/actions/meta-onboarding.ts`, after `subscribeWaba`:
  1. GET `/{phoneNumberId}?fields=status` (via `metaRequest`). If `status === 'CONNECTED'`
     → `updateMetaAccountRegistration('connected')` → return `{ success:true, status:'connected' }`
     (idempotent: never re-register an already-active number).
  2. Else generate a 6-digit PIN (`crypto.randomInt(100000, 1000000)`), then
     `registerPhoneNumber(bisuat, phoneNumberId, pin)`:
     - success → persist pin encrypted + status `connected` → `{ success:true, status:'connected' }`.
     - `catch (MetaGraphApiError)` map (see PLAYBOOK):
       - `errorSubcode === 2388001` → status `needs_2sv` →
         msg: "Tu número tiene verificación en dos pasos de un proveedor anterior. Desactívala
         en WhatsApp Manager (Two-step verification → Turn off, confirmas por email) y reintenta."
       - message matches `/payment method|Cannot Migrate Phone Number/i` → status `needs_payment` →
         msg: "Tu cuenta de WhatsApp necesita un método de pago. Agrégalo en WhatsApp Manager →
         Payment settings y reintenta."
       - else → status `register_failed` → generic retry msg (detail stays server-side).
- Return type becomes
  `{ success:true; status:'connected'|'needs_2sv'|'needs_payment'|'register_failed'; message?:string }
   | { success:false; error:string }`.
  The connection ROW already exists (subscribe succeeded) — register problems are surfaced as
  `status` + `message`, NOT as `success:false` (the number IS connected, just not yet active).
- Never log the code, plaintext BISUAT, or the PIN (existing T-38 security contract).

### T3 — UI: surface the activation status (minimal, non-deferred slice)
- `src/components/settings/connect-whatsapp.tsx`: on action result, if
  `success && status !== 'connected'` → show `message` as a warning toast (actionable);
  if `connected` → success toast. (Full "Conectado" panel + disconnect stays the Plan 05
  deferred follow-up — out of scope here.)

### T4 — Tests (Wave 0 parity — extend existing meta suite)
- `src/lib/meta/__tests__/embedded-signup.test.ts` (or a new `meta-onboarding` unit test with
  `fetch`/`metaRequest` mocked): assert the error→status mapping:
  - 2388001 → `needs_2sv`; "Cannot Migrate Phone Number ... payment method" → `needs_payment`;
    `{ success:true }` → `connected`; already-CONNECTED status short-circuits (no register call).
- Run `npx vitest run src/lib/meta/ src/app/api/webhooks/meta/` → all green.
