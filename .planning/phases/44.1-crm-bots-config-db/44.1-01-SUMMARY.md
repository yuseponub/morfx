## End-to-End Verification (Task 9 Parte A)

**Contexto:** el baseline pre-deploy (antes de Phase 44.1) era: sin env vars seteadas en Vercel, `process.env.CRM_BOT_ENABLED !== 'false'` (undefined) → handler procedia → bots activos. Nunca hubo kill-switch real. Post-deploy: kill-switch ahora es real, flipable via SQL.

### Ciclo de verificacion

**Paso 1 — Baseline (`crm_bot_enabled=true` en DB):**
```bash
curl -i -X POST https://morfx.app/api/v1/crm-bots/reader ...
```
- Respuesta: `HTTP/1.1 200 OK`
- Latencia: 7s (reader con LLM)
- **OK.** Baseline establecido.

**Paso 2 — Flip kill-switch OFF:**
```sql
UPDATE platform_config SET value = 'false'::jsonb WHERE key = 'crm_bot_enabled';
```
Esperar 35s (>TTL de 30s). Curl de nuevo:
- Respuesta: `HTTP/1.1 200 OK`
- **BUG.** Expected 503. Kill-switch no disparo.

**Paso 3 — Root cause investigation:**

Query directo desde WSL al REST API de Supabase usando `SUPABASE_SERVICE_ROLE_KEY`:
```bash
curl -s "$SUPABASE_URL/rest/v1/platform_config?key=eq.crm_bot_enabled" \
  -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"
```
Respuesta:
```json
{"code":"42501","message":"permission denied for table platform_config"}
```

**Root cause real:** cuando la tabla se creo via Supabase Studio SQL Editor (Task 2 checkpoint), el `service_role` NO fue auto-granted privilegios sobre la tabla. Phase 44 (tabla `crm_bot_actions`) tuvo suerte porque fue creada via el path correcto (que auto-grant-ea); nuestra migration 44.1 se aplico via raw SQL Editor. Resultado: `getPlatformConfig` hit un 42501, el fail-open catch lo enmascaro y retornaba `true` (fallback) — el kill-switch **nunca podia disparar en produccion**.

**Paso 4 — Fix aplicado (via Supabase SQL Editor):**
```sql
GRANT ALL    ON TABLE public.platform_config TO service_role;
GRANT SELECT ON TABLE public.platform_config TO authenticated;
```

**Paso 5 — Re-test post-fix (crm_bot_enabled=false):**
- Respuesta: `HTTP/1.1 503 Service Unavailable` body `{"error":"CRM bots globally disabled","code":"KILL_SWITCH","retryable":false}`
- Latencia: 1.4s (cache miss → DB read → 503)
- **PASS.**

**Paso 6 — Revert + re-verify:**
```sql
UPDATE platform_config SET value = 'true'::jsonb WHERE key = 'crm_bot_enabled';
```
Esperar 35s. Curl:
- Respuesta: `HTTP/1.1 200 OK`
- Latencia: 7.1s (normal reader)
- **PASS.** Round-trip verificado.

**Paso 7 — Persistencia del fix:**
El GRANTs solo estaba en la DB de produccion, no en el archivo de migration. Commit `ac4b6b8` agrega los GRANTs in-file para que futuros ambientes (staging/dev/nueva prod) no hereden el bug.
