<mp4_scan_spec>
`isAudioOnlyMp4(buf: Buffer): boolean` — pure container heuristic.

Behavior:
- Returns `true`  ⇔ the buffer contains an mp4/quicktime `'soun'` (audio) handler marker AND does NOT contain a `'vide'` (video) handler marker.
- Returns `false` in every other case: a `'vide'` marker is present (real video track), OR no `'soun'` marker (not an audio mp4 we recognize), OR the buffer is too small / garbage / not an mp4.

Implementation (defensive, bounded):
- Guard inputs: if `!Buffer.isBuffer(buf)` or `buf.length < 8` → return `false`.
- Bound the scan: only inspect the first `Math.min(buf.length, 524288)` bytes (512KB). NEVER scan unboundedly (untrusted uploaded buffer — see threat model).
- Search the bounded slice for the ASCII markers `'vide'` and `'soun'` (these are the `handler_type` field values inside an `hdlr` box in the `moov`/`trak`/`mdia` hierarchy). Use `slice.indexOf('vide', 0, 'ascii')` / `slice.indexOf('soun', 0, 'ascii')` (or `Buffer.from('vide','ascii')`).
- Decision: `if (videIdx !== -1) return false; if (sounIdx !== -1) return true; return false;`
- Wrap the body so it NEVER throws on unexpected input (return `false` on any caught error). Keep it a pure function — no console, no I/O.

Why markers (not full box parsing): the `hdlr` `handler_type` literally contains the 4-byte ASCII `vide`/`soun`. Scanning for these tokens within the bounded moov region is robust for the chat-export `.mp4` and Android voice-note `.mp4`/`.m4a` cases, and is defensive against malformed boxes (a strict box walker could throw on truncated atoms). The presence of `vide` is authoritative for "has a video track"; its absence + presence of `soun` is authoritative for "audio-only".
</mp4_scan_spec>
