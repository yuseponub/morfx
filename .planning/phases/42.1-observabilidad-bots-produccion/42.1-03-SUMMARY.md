# Phase 42.1 Plan 03: Fetch Wrapper + Admin Client Refactor Summary

## Outcome

Wave 2 Plan 03 completo. Existe ahora un punto unico no invasivo para capturar TODAS las queries SQL del repo: cualquier llamada a `createAdminClient()` que se ejecute dentro de un `runWithCollector(...)` emite eventos `ObservabilityQuery` automaticamente. Cuando no hay collector activo (flag OFF, codigo fuera del pipeline de agentes, scripts, etc.) el wrapper hace fast-path al fetch original — overhead = una llamada de funcion arrow. Domain layer, server actions, webhooks e Inngest functions siguen funcionando sin un solo cambio.

El stub para Anthropic existe ya en el wrapper (registra event `tool_call/ai_call_raw`) pero la captura completa (token counts, prompt version hash, cost, response content) la implementa Plan 04 reemplazando ese branch con `recordAiCall`.

## Diff Conceptual de admin.ts

**Antes (legacy):**

```typescript
export function createAdminClient() {
  return createSupabaseClient(url, serviceRoleKey, {
    auth: { autoRefreshToken: false, persistSession: false },
  })
}
```

Un solo export, sin instrumentation, sin separacion entre uso interno/externo.

**Despues (Plan 03):**

```typescript
function readEnv(): { url: string; key: string } { /* validacion compartida */ }

// DEFAULT — usado por todo el repo (~30 archivos importan esta).
// Identico al legacy excepto por global.fetch wrappeado.
export function createAdminClient(): SupabaseClient {
  const { url, key } = readEnv()
  return createSupabaseClient(url, key, {
    auth: { autoRefreshToken: false, persistSession: false },
    global: { fetch: makeObservableFetch(fetch, 'supabase') }, // <-- unica diferencia
  })
}

// EXCLUSIVO para src/lib/observability/* — sin instrumentation,
// previene recursion infinita del flush sobre sus propias tablas.
export function createRawAdminClient(): SupabaseClient {
  const { url, key } = readEnv()
  return createSupabaseClient(url, key, {
    auth: { autoRefreshToken: false, persistSession: false },
  })
}
```

## Anti-Recursion: Por que `createRawAdminClient`

Cuando Plan 07 implemente `collector.flush()`, hara algo como:

```typescript
async flush() {
  const supabase = createRawAdminClient() // ← CRITICO
  await supabase.from('agent_observability_queries').insert(this.queries.map(...))
}
```

Si usaramos `createAdminClient()` (instrumentado), el INSERT pasaria por `makeObservableFetch`, que detectaria el collector activo, llamaria `recordQuery` para anadir UNA NUEVA query al collector, modificaria `this.queries` mientras estamos iterando sobre ella, y al proximo flush habria mas queries por flushear → loop infinito + connection pool exhaustion + stack overflow. La separacion explicita en dos factories hace este bug imposible.

**WARNING:** `createRawAdminClient` NO debe usarse fuera de `src/lib/observability/*`. Esta documentado en el jsdoc del export. Verificacion futura sugerida: ESLint rule `no-restricted-imports` o un grep gate en CI.

## Deviations from Plan

**1. [Rule 3 - Type alignment] Adapted error path to existing collector API**

- **Found during:** Task 1 implementation
- **Issue:** El plan describia `collector.recordError({ kind, url, method, error: err, durationMs })`, pero el `RecordedErrorInfo` interface existente en collector.ts (Plan 02) es `{ name, message, stack? }`.
- **Fix:** En el catch convertimos `err` a `Error`, llamamos `recordError({ name, message, stack })` con la shape correcta, y AGREGAMOS un `recordQuery`/`recordEvent` fallido segun kind para preservar la informacion contextual (url, method, statusCode=0) en el bag correspondiente. Esto es estrictamente mejor que el plan: el timeline muestra DONDE murio el turno, no solo QUE error fue.
- **Files modified:** src/lib/observability/fetch-wrapper.ts
- **Commit:** aa174fd

**2. [Rule 2 - Defensive] URL parsing con try/catch silencioso**

- **Found during:** Task 1 implementation
- **Issue:** `new URL(url)` puede throw si la url esta malformada (raro, pero posible con tests o casos edge). El plan no especificaba comportamiento.
- **Fix:** try/catch alrededor de `new URL`, fallback a `tableName='unknown'`, sin re-throw. REGLA 6 (proteger prod agent path).
- **Files modified:** src/lib/observability/fetch-wrapper.ts
- **Commit:** aa174fd

**3. [Rule 2 - Completeness] PUT operation mapeado a 'upsert'**

- **Found during:** Task 1 implementation
- **Issue:** El ejemplo del research mapeaba GET/POST/PATCH/DELETE pero ignoraba PUT. `ObservabilityQuery['operation']` (Plan 02) incluye `'upsert'`. Supabase-js usa PUT para upserts.
- **Fix:** Agregado `upper === 'PUT' ? 'upsert' : ...` en la cadena de mapping.
- **Files modified:** src/lib/observability/fetch-wrapper.ts
- **Commit:** aa174fd

Ningun otro deviation. Domain layer 100% intacto.
