# Phase 999.1 Plan 04: Composer UI del mensaje interactivo Summary

Construye la UI del operador para componer y enviar mensajes interactivos de WhatsApp (botones | lista). Tres componentes nuevos + el cableado del toolbar: (1) `interactive-bubble.tsx` — render PURO estilo WhatsApp (header/body/footer + afordancia botones|lista), single source of truth reutilizable por el preview en vivo Y la burbuja outbound de Plan 05; (2) `interactive-composer-modal.tsx` — el builder Dialog con toggle Botones|Lista, contadores inline + validacion COMPLETA de limites Meta (D-05a) que gatea el boton Enviar, preview en vivo via `InteractiveBubble` (D-05b), y envio via la server action `sendInteractiveMessage` con manejo amistoso del error de lista 360dialog (Regla 3 / UI-SPEC Area 4); (3) `interactive-composer-button.tsx` — el trigger del toolbar con forma de icono ghost; y el cableado en `message-input.tsx` insertando el boton SOLO en la rama de ventana abierta (D-02 — la ubicacion ES el gate, nunca en la rama cerrada). Los ids de boton/fila son auto-generados deterministas y unicos, nunca tecleados por el operador (D-05/id). Sin `dangerouslySetInnerHTML` (T-999.1-XSS). Las 29 pruebas del fase quedan GREEN (Regla 6).

## What Was Built

### Task 1 — `interactive-bubble.tsx` (render puro compartido) — commit `b603f299`
- `'use client'` render PURO (sin state, sin efectos). Exporta `InteractiveContent` (matchea el JSONB almacenado: `interactiveType`, `body`, `header?`, `footer?`, `buttons?`, `buttonLabel?`, `sections?`) + `InteractiveBubble({ content })`.
- Extiende el shell de `whatsapp-bubble.tsx` VERBATIM (verdes `#d9fdd3`/`#005c4b` + `text-black dark:text-white` LOCKED por UI-SPEC Color).
- Afordancia interactiva (UI-SPEC Area 3): `buttons` → stack vertical, `border-t` separador, `text-[#00a5f4] dark:text-[#53bdeb]` link-blue, centrado; `list` → una fila "list-open" con icono `List` + `buttonLabel`, mismo link-blue + border-t.
- Cero `dangerouslySetInnerHTML` (XSS gate == 0). El comentario que mencionaba la palabra se reescribio para mantener el grep en 0 (mismo patron documentado en Plan 03).

### Task 2 — `interactive-composer-modal.tsx` + `interactive-composer-button.tsx` — commit `50a73bdb`
- **Modal:** props `{ open, onOpenChange, conversationId, contactPhone, onSend? }`. Mirror de `template-send-modal.tsx` (Dialog `max-w-2xl` + `ScrollArea` + `DialogFooter` + reset-on-open via `useEffect([open])` + `handleSend` async + `Loader2` + `toast`).
  - **Estado:** `interactiveType` (default `'buttons'`), `header`/`body`/`footer`, `buttons` (default `[{id:'btn_0',title:''}]`), `buttonLabel`, `sections` (default `[{title:'',rows:[{id:'row_0_0',title:''}]}]`), `sending`.
  - **ids auto (D-05/id):** helpers `reindexButtons` (`btn_${i}`) y `reindexSections` (`row_${si}_${ri}`) re-aplicados en cada add/remove → siempre unicos; el operador nunca ve ni edita ids.
  - **Validacion cada render:** `validateButtons` | `validateList` segun el toggle → `hasErrors` gatea `disabled={sending || hasErrors}` en el boton Enviar (D-05a). Errores agregados se listan en `text-[11px] text-destructive` arriba del footer; cada campo tiene contador en vivo (subcomponente `Field`) que vira a `text-destructive` + mensaje "Máx N caracteres" cuando excede.
  - **Preview en vivo (D-05b):** construye `InteractiveContent` desde el estado y lo renderiza con `<InteractiveBubble content={previewContent} />`.
  - **Envio:** payload discriminado por tipo → `sendInteractiveMessage(conversationId, payload)`; `'error' in result` → toast (el caso `'lista no soportada en 360dialog (legacy)'` se traduce al mensaje amistoso provider-agnostic, Dialog queda abierto); exito → `toast.success` + `onOpenChange(false)` + `onSend?.()`.
  - **Tabs Botones:** hasta 3 botones, `X` para quitar cuando `>1`, `Agregar botón` oculto en 3. **Tabs Lista:** `buttonLabel` + hasta 10 secciones con titulo opcional + 1-N filas (titulo + descripcion opcional + `X`), `Agregar fila` / `Agregar sección`.
  - **Footer primario** estilo editorial: `bg-[var(--ink-1)] text-[var(--paper-0)] hover:bg-[var(--ink-2)] rounded-[4px]`.
- **Button:** `'use client'` mirror de `template-button.tsx` con la forma de icono ghost del boton adjuntar (`variant="ghost" size="icon" h-10 w-10 flex-shrink-0`), icono `List`, `title="Mensaje interactivo"`, `aria-label="Abrir compositor de mensaje interactivo"`. Renderiza `InteractiveComposerModal`.

## Deviations from Plan

None — plan executed exactly as written. Tres notas menores de implementacion (no son deviations de comportamiento):
1. **Subcomponente `Field`** extraido localmente (label + contador en vivo + mensaje de exceso) para no repetir el patron de contador en ~10 campos. El UI-SPEC describe el contrato del contador campo-por-campo; el subcomponente lo cumple identico.

3. **`contactPhone` recibido como `_contactPhone`** en el modal (prefijo `_` para el lint de no-unused) — el modal lo acepta por contrato/parity con el button y el server action lo resuelve internamente desde la conversacion; no se usa en el cliente.
