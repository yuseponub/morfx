## Summary

Research confirms the user's hypothesis with HIGH confidence: **a domain that only exposes `/login` is a legitimate and common cause of Meta Business Verification rejection**, though it's rarely the sole cause listed by partners. Meta's documented primary rejection reasons are (1) document/website name mismatches, (2) illegible or wrong-language documents, (3) domain not verified in Meta Business Manager, and (4) "website looks inactive / doesn't represent the business" — which is exactly the failure mode for morfx.app today. Meta reviewers need to land on a public URL and see, without credentials: legal business name, logo, products description, physical address, phone, email, and functioning privacy/terms links. All these must match what was submitted in the Cámara de Comercio and RUT.

The Next.js stack is **Next.js 16.1.6** (not 15 — verified from `package.json`), which renamed `middleware.ts` to `proxy.ts` but still accepts `middleware.ts` with a deprecation warning. The codebase already has `middleware.ts` wrapping `updateSession()` from `@/lib/supabase/middleware`, and that `updateSession()` hard-redirects everything not in a small whitelist to `/login`. This is the single biggest blocker and must be fixed before anything else works. Additionally, `src/app/page.tsx` explicitly redirects to `/login` or `/crm` — it must be replaced with the landing.

For bilingual ES/EN with ES as default and toggle in header, **next-intl 4.x with `localePrefix: 'as-needed'`** is the standard — `/` and `/privacy` serve Spanish, `/en` and `/en/privacy` serve English. This requires a `[locale]` route group for public routes only; authenticated routes (`/crm`, `/dashboard`, etc.) stay outside the locale segment (they are single-language ES). For privacy/terms, do NOT auto-generate via Termly/iubenda for this case — their Law 1581 coverage is unverified. Write the documents from a curated Colombian-lawyer template and mark them clearly "Versión 1.0 – 2026-04-14" with a flag for legal review post-Meta-approval.

**Primary recommendation:** Restructure routes as `src/app/(marketing)/[locale]/` with three routes (`/`, `/privacy`, `/terms`), add these paths to the `publicRoutes` whitelist in `src/lib/supabase/middleware.ts:38-46`, replace `src/app/page.tsx` with a marketing hero, install `next-intl`, and add Meta Business Manager DNS TXT domain verification to the pre-resubmission checklist.

## Architecture Patterns

### Recommended Project Structure

Route groups separate public marketing (bilingual) from authed app (Spanish-only). This is the idiomatic Next.js App Router pattern.

```
src/app/
├── (marketing)/                    # NEW — public routes, no auth
│   └── [locale]/                   # next-intl dynamic segment
│       ├── layout.tsx              # Marketing shell: header (lang/theme/login), footer
│       ├── page.tsx                # Landing hero + features + contact
│       ├── privacy/
│       │   └── page.tsx            # Privacy policy (ES or EN based on [locale])
│       └── terms/
│           └── page.tsx            # Terms of service
├── (auth)/                         # EXISTING — login, signup, etc. (no locale)
│   ├── layout.tsx
│   ├── login/page.tsx
│   ├── signup/page.tsx
│   └── ...
├── (dashboard)/ OR /crm/, /agentes/, etc.  # EXISTING — authed app (no locale)
├── layout.tsx                      # Root — keep ThemeProvider, Toaster, fonts
└── page.tsx                        # DELETE or convert to redirect to (marketing)/[locale]

messages/                           # NEW — translation files at project root
├── es.json
└── en.json

src/i18n/                           # NEW
├── routing.ts                      # defineRouting({ locales: ['es','en'], defaultLocale:'es', localePrefix:'as-needed' })
├── request.ts                      # getRequestConfig for Server Components
└── navigation.ts                   # createNavigation(routing) → typed Link, redirect

middleware.ts                       # EXISTING — modify to skip marketing routes + handle next-intl
                                    # (In Next.js 16 can rename to proxy.ts, but middleware.ts still works)
```

**Key decisions:**

1. **Locale only applies to `(marketing)` routes.** Dashboard, CRM, agents are Spanish-only — no `[locale]` segment for them. This avoids forcing every authed route to deal with i18n when it's not a product requirement.
2. **`localePrefix: 'as-needed'`** — default locale (ES) has no URL prefix, so `/` = Spanish landing. `/en` = English landing. This is SEO-friendly and matches user decision "español por defecto."
3. **Marketing layout** renders the public shell (header with language toggle, theme toggle, "Iniciar sesión" link, and "Ir al dashboard" link conditionally if user is authed; footer with legal data, links). Does NOT call Supabase for auth requirement, but MAY read session to show "Ir al dashboard" vs "Iniciar sesión."
4. **Root `src/app/page.tsx` is deleted.** With `(marketing)/[locale]/page.tsx` and `localePrefix: 'as-needed'`, next-intl's middleware serves `/` from `/es/` internally without needing a root page.

### Pattern 1: Public route whitelist in middleware

**What:** Supabase middleware must NOT redirect `/`, `/privacy`, `/terms`, `/en`, `/en/privacy`, `/en/terms` to `/login`.

**When to use:** Always. This is the #1 blocker. Currently `src/lib/supabase/middleware.ts:38-46` has:

```typescript
const publicRoutes = [
  '/login', '/signup', '/forgot-password', '/reset-password',
  '/auth/callback', '/auth/confirm', '/invite',
]
```

**Fix:** Add marketing roots + bilingual paths AND use `startsWith` matching for locale prefix:

```typescript
// Source: analyzed src/lib/supabase/middleware.ts:38-53
const publicRoutes = [
  '/login', '/signup', '/forgot-password', '/reset-password',
  '/auth/callback', '/auth/confirm', '/invite',
]
const publicPathPrefixes = ['/privacy', '/terms', '/en/privacy', '/en/terms']
const publicExactPaths = ['/', '/en']

const pathname = request.nextUrl.pathname
const isPublicRoute =
  publicExactPaths.includes(pathname) ||
  publicRoutes.includes(pathname) ||
  publicPathPrefixes.some(p => pathname === p || pathname.startsWith(p + '/')) ||
  pathname.startsWith('/auth/') ||
  pathname.startsWith('/invite/')
```

### Pattern 2: next-intl proxy (or middleware) composition with Supabase

**What:** Run next-intl's locale detection BEFORE Supabase session check for marketing routes; run Supabase middleware for everything else.

**Why:** next-intl rewrites/redirects based on locale (e.g., browser sends `Accept-Language: en` → redirects to `/en`). If Supabase runs first and redirects to `/login`, next-intl never executes.

**Example:**

```typescript
// Source: adapted from next-intl docs + existing middleware.ts
import createIntlMiddleware from 'next-intl/middleware'
import { routing } from '@/i18n/routing'
import { updateSession } from '@/lib/supabase/middleware'
// ... other imports unchanged

const intlMiddleware = createIntlMiddleware(routing)

const MARKETING_PATHS = ['/', '/privacy', '/terms']
function isMarketingPath(pathname: string) {
  // Strip /en prefix
  const stripped = pathname.replace(/^\/en(?=\/|$)/, '') || '/'
  return MARKETING_PATHS.some(p => stripped === p || stripped.startsWith(p + '/'))
}

export async function middleware(request: NextRequest) {
  const pathname = request.nextUrl.pathname

  // Webhook/inngest/manychat/mobile/api-tools bypasses (unchanged)
  if (pathname.startsWith('/api/webhooks')) return NextResponse.next()
  // ... other bypasses

  // Marketing routes → next-intl handles locale, no Supabase redirect
  if (isMarketingPath(pathname)) {
    return intlMiddleware(request)
  }

  // Everything else → existing Supabase session flow
  return await updateSession(request)
}
```

### Pattern 3: Marketing layout with optional auth awareness

**What:** Marketing layout reads Supabase session OPTIONALLY — if user is authed, show "Ir al dashboard"; otherwise show "Iniciar sesión." Does NOT redirect either way.

**Example:**

```typescript
// src/app/(marketing)/[locale]/layout.tsx
import { NextIntlClientProvider } from 'next-intl'
import { setRequestLocale } from 'next-intl/server'
import { hasLocale } from 'next-intl'
import { notFound } from 'next/navigation'
import { routing } from '@/i18n/routing'
import { createClient } from '@/lib/supabase/server'
import { MarketingHeader } from '@/components/marketing/header'
import { MarketingFooter } from '@/components/marketing/footer'

export function generateStaticParams() {
  return routing.locales.map(locale => ({ locale }))
}

export default async function MarketingLayout({
  children,
  params,
}: {
  children: React.ReactNode
  params: Promise<{ locale: string }>
}) {
  const { locale } = await params
  if (!hasLocale(routing.locales, locale)) notFound()
  setRequestLocale(locale)

  // Optional auth awareness (no redirect)
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  return (
    <NextIntlClientProvider>
      <div className="min-h-screen flex flex-col">
        <MarketingHeader isAuthed={!!user} />
        <main className="flex-1">{children}</main>
        <MarketingFooter />
      </div>
    </NextIntlClientProvider>
  )
}
```

### Anti-Patterns to Avoid

- **Putting marketing under `[locale]` at app root.** Would force `/crm`, `/agentes`, etc. into `[locale]/crm`, `[locale]/agentes`. Breaks existing authed routes. Keep `[locale]` scoped to `(marketing)` route group only.
- **Writing translations inline as `locale === 'es' ? 'Hola' : 'Hello'`.** Quick but unmaintainable past 10 strings. Use next-intl's `useTranslations('Namespace')` even for 3 pages.
- **Using cookie-based locale without URL prefix.** Breaks SEO (Google indexes one language), breaks linkable translation-specific URLs for Meta reviewer. Use `localePrefix: 'as-needed'`.
- **Redirecting `/` to `/es`.** Never do this — it creates an indexable redirect that Meta reviewer might see. `localePrefix: 'as-needed'` serves ES content AT `/` directly.
- **Deleting `src/app/page.tsx` without checking build.** Next.js may still look for a root `page.tsx`. Safer to keep it as a minimal redirect or empty page that delegates to the marketing segment. TEST LOCALLY.

## i18n Approach

### Recommendation: next-intl 4.x with `localePrefix: 'as-needed'`

**Why not hand-rolled:** 3 pages × 2 languages × ~30 strings/page = ~180 strings. Hand-rolling means scattered ternaries, inline strings, no TypeScript safety, painful to extend. next-intl adds 1 dependency and solves it.

**Why not i18next:** Works, but official Next.js i18n guide and most 2025-2026 tutorials recommend next-intl for App Router. No reason to pick the less-standard option here.

**Why `as-needed` not `always`:** User decision — ES is default, no `/es` prefix desired (clean URLs for Meta reviewer). `as-needed` gives `/` = ES, `/en` = EN.

## Domain Verification Procedure

Meta Business Manager domain verification is SEPARATE from Business Verification (company-level). It's a strong signal that you own the domain and should be done before resubmitting.

### Steps (DNS TXT method — recommended for Porkbun)

Source: Meta Business Help Center + Qtonix + Buffer docs, cross-verified.

1. Open **Meta Business Suite** → **Business Settings** (or Configuración del negocio)
2. Left sidebar → **Brand Safety** (Seguridad de marca) → **Domains** (Dominios)
3. Click **Add** / **Agregar** → enter `morfx.app` → click **Add**
4. In the dialog, select **DNS Verification** tab
5. Copy the TXT record value (looks like `facebook-domain-verification=abc123xyz...`)
6. Log into **Porkbun** (domain registrar) → Domain Management → `morfx.app` → DNS Records
7. Add new record:
   - Type: **TXT**
   - Host: blank (or `@`)
   - Answer: `facebook-domain-verification=abc123xyz...` (paste exactly)
   - TTL: 600 (default)
8. Wait 5-30 minutes for DNS propagation (can check with `dig TXT morfx.app` or dnschecker.org)
9. Back in Meta Business Suite → click **Verify** at the bottom of the DNS Verification tab
10. Status should flip to **Verified** ✓

**Alternative methods (if DNS fails):**
- **Meta tag method** — add `<meta name="facebook-domain-verification" content="abc123xyz..." />` to the root `<head>` in `src/app/layout.tsx`, deploy, click Verify
- **HTML file upload** — Meta gives you a file like `abc123.html`, place it in `/public/`, deploy, click Verify (Vercel serves `/public/*` at domain root)

**Gotcha:** Once verified, TXT record CAN be removed without losing verification status (per Meta docs). But recommend leaving it for re-verification safety.

**Timing:** Do this BEFORE resubmitting Business Verification. It's a prerequisite signal per Graphed and Delightchat partner blogs.

## Privacy / Terms Template Sources

### Recommendation: Hand-write from Colombian/EU curated sources, NOT auto-generator

**Why not Termly:**
- HIGH confidence Termly generates GDPR-compliant English policies
- LOW confidence Termly has Ley 1581 / Habeas Data coverage — no partner source verified this
- Termly auto-translate is browser-language based; doesn't integrate with Next.js route-based locale

**Why not iubenda:**
- Similar: GDPR strong, Ley 1581 unverified
- Additional languages cost +$6.75/each

## Common Pitfalls

### Pitfall 1: Supabase middleware blocks marketing routes

**What goes wrong:** Landing loads in dev but in production `/privacy` redirects to `/login`. Meta reviewer sees 302 → login and marks business as unverifiable.

**Why it happens:** `src/lib/supabase/middleware.ts:48-60` redirects ALL non-whitelisted routes to `/login` if no user session. Without adding marketing paths to `publicRoutes`, they're treated as authed routes.

**How to avoid:** 
1. Add `/`, `/en`, `/privacy`, `/terms`, `/en/privacy`, `/en/terms` to whitelist
2. Use prefix matching (`startsWith`) for `/privacy` and `/terms` to catch `/privacy/`, `/terms/`
3. Test in production (Vercel preview) before assuming local dev behavior applies

**Warning signs:** Incognito window visiting `morfx.app` in production doesn't show landing → STOP, middleware bug.

### Pitfall 2: next-intl middleware conflicts with Supabase middleware

**What goes wrong:** Only one `middleware.ts` file allowed per project. If both run naively, next-intl's redirects fight Supabase's redirects → infinite loop or wrong page.

**Why it happens:** Both want to inspect `pathname` and potentially redirect.

**How to avoid:** Compose them explicitly — see Pattern 2 above. Marketing paths use `intlMiddleware(request)`, everything else uses `updateSession(request)`. No overlap.

**Warning signs:** `ERR_TOO_MANY_REDIRECTS` in browser, or visiting `/en` redirects to `/login` → check middleware composition.

### Pitfall 3: Root `src/app/page.tsx` vs `(marketing)/[locale]/page.tsx` conflict

**What goes wrong:** Next.js tries to resolve `/` and gets confused if there's both a root `page.tsx` AND a `[locale]/page.tsx` that next-intl rewrites to.

**Why it happens:** Route collision. Next.js can't have two routes that both match `/`.

**How to avoid:** DELETE `src/app/page.tsx`. next-intl's middleware rewrites `/` internally to `[locale]/page.tsx` with `locale='es'`. No physical root page needed.

**Warning signs:** Build error `Duplicate route` or visiting `/` returns 404.

### Pitfall 4: Domain verification done AFTER Business Verification resubmit

**What goes wrong:** Resubmit Business Verification, wait 2-14 days, get rejected again because domain wasn't verified.

**Why it happens:** These are SEPARATE processes in Meta Business Manager. Business Verification reviewers look at domain verification status as a trust signal.

**How to avoid:** Complete domain DNS TXT verification FIRST. Wait for status = Verified ✓. THEN resubmit Business Verification.

**Warning signs:** In Meta Business Suite → Brand Safety → Domains, morfx.app status says "Pending" or not listed.

### Pitfall 5: Legal business name missing from visible content

**What goes wrong:** Landing has "morfx" everywhere (product brand) but never "MORFX S.A.S." (legal entity). Reviewer can't connect website to CC documents.

**Why it happens:** Product branding ≠ legal entity. Designer instinct is to use clean product name.

**How to avoid:** 
- Include "MORFX S.A.S." prominently at least twice: once in "About" section body, once in footer beside NIT
- Format: `MORFX S.A.S. — NIT 902052328-5`
- Include the textual objeto social paraphrased in About section

**Warning signs:** `grep -r "MORFX S.A.S." src/app/(marketing)` returns 0 matches → NOT VERIFIABLE.

### Pitfall 6: Address format mismatch between website and documents

**What goes wrong:** Website shows "Cra 38 #42-17 Apto 1601B". Documents submitted say "Carrera 38 # 42 - 17 Apartamento 1601B". Reviewer flags as mismatch.

**Why it happens:** Humans write shortcuts. Meta pattern-matches literally.

**How to avoid:** Copy address from CC/RUT VERBATIM. Exact string: `Carrera 38 # 42 - 17 Apartamento 1601B, Bucaramanga, Santander, Colombia`. Both ES and EN pages show this exact Spanish string (don't translate "Carrera" or "Apartamento").

**Warning signs:** Address differs in any character from CC document → TEST BEFORE SUBMIT.

### Pitfall 7: Metadata not localized

**What goes wrong:** Page title is "morfx" in both languages. OG image preview in WhatsApp/Facebook shows Spanish title for English page.

**Why it happens:** `export const metadata` is static; doesn't know current locale.

**How to avoid:** Use `generateMetadata({ params })` with locale-aware titles:

```typescript
// Source: next-intl docs

  const { locale } = await params
  const t = await getTranslations({ locale, namespace: 'Metadata' })
  return { title: t('title'), description: t('description') }
}
```

**Warning signs:** Test by viewing `view-source:` of `/` vs `/en`, titles should differ.

### Pitfall 8: Email link to morfx.colombia@gmail.com vs business email

**What goes wrong:** Partner sources (Delightchat #12) flag personal Gmail as a weak signal for verification. Not a hard blocker but reduces confidence.

**Why it happens:** Business registered recently, no custom email domain yet.

**How to avoid:** Either (a) set up `contacto@morfx.app` via Porkbun email forwarding or Google Workspace before resubmit, OR (b) accept the risk and use Gmail — note it in the "Open Questions" section for the planner.

**Warning signs:** N/A — this is a risk-accept decision, not a fail-fast.

### Example 4: Composed middleware (Supabase + next-intl)

```typescript
// middleware.ts — PROJECT ROOT (replaces existing)
// Source: existing middleware.ts + next-intl docs, composed
import { type NextRequest, NextResponse } from 'next/server'
import createIntlMiddleware from 'next-intl/middleware'
import { routing } from '@/i18n/routing'
import { updateSession } from '@/lib/supabase/middleware'
import { validateApiKey, extractApiKey } from '@/lib/auth/api-key'

const intlMiddleware = createIntlMiddleware(routing)

const MARKETING_PREFIXES = ['/privacy', '/terms']
function isMarketingPath(pathname: string): boolean {
  // Strip /en prefix if present
  const stripped = pathname.replace(/^\/en(?=\/|$)/, '') || '/'
  if (stripped === '/') return true
  return MARKETING_PREFIXES.some(
    p => stripped === p || stripped.startsWith(p + '/')
  )
}

export async function middleware(request: NextRequest) {
  const pathname = request.nextUrl.pathname

  // ==================== API BYPASSES (unchanged) ====================
  if (pathname.startsWith('/api/webhooks')) return NextResponse.next()
  if (pathname.startsWith('/api/inngest')) return NextResponse.next()
  if (pathname.startsWith('/api/manychat')) return NextResponse.next()
  if (pathname.startsWith('/api/mobile')) return NextResponse.next()
  if (pathname.startsWith('/api/temp-send-agendados')) return NextResponse.next()
  if (pathname.startsWith('/api/v1/tools')) {
    // ... existing API key logic unchanged
    const authHeader = request.headers.get('authorization')
    const apiKey = extractApiKey(authHeader)
    if (!apiKey) return NextResponse.json({ error: 'Missing API key', code: 'MISSING_API_KEY' }, { status: 401 })
    const validation = await validateApiKey(apiKey)
    if (!validation.valid) return NextResponse.json({ error: validation.error || 'Invalid API key', code: 'INVALID_API_KEY' }, { status: 401 })
    const response = NextResponse.next()
    response.headers.set('x-workspace-id', validation.workspaceId!)
    response.headers.set('x-permissions', JSON.stringify(validation.permissions || []))
    response.headers.set('x-api-key-prefix', apiKey.substring(0, 8))
    return response
  }

  // ==================== MARKETING (PUBLIC) ====================
  if (isMarketingPath(pathname)) {
    return intlMiddleware(request)
  }

  // ==================== SESSION AUTH (everything else) ====================
  return await updateSession(request)
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```
