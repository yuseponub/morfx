### Existing patterns to leverage

- **`createAdminClient()`** — canonical server-side DB accessor. Used throughout domain layer. Already imported in many places.
- **Domain layer shape** — see `src/lib/domain/tags.ts` for smallest canonical example of `DomainResult<T>` discriminated union and workspace-scoped filter pattern. Our new `platform-config.ts` module does NOT filter by workspace (platform-level config) — so it deviates from domain layer convention in that ONE dimension. Document the deviation.
- **Time-based cache** — not a widespread pattern in this codebase. Grep for `setInterval` and `Date.now()` shows no existing LRU/TTL cache module. Hand-roll a minimal one (Map<string, {value, expiresAt}>) in the new `platform-config.ts`.

## Pitfalls

### Pitfall 1 — Async cascade from converting `rateLimiter.check`

The rate-limiter is called synchronously in ~5 places (3 crm-bots routes + 1 tools executor + 1 test). Converting to async would force `await` on every call site. **Mitigation:** keep `rateLimiter.check` sync. Route callers resolve the limit via `await getPlatformConfig('crm_bot_rate_limit_per_min', 50)` and pass it as an explicit override param. Existing DEFAULTS record stays as fallback for non-route callers.

### Pitfall 2 — Per-request env read (inherited from Phase 44)

Phase 44 explicitly reads `CRM_BOT_ENABLED` INSIDE the handler (not at module load) so that Vercel env refresh doesn't require the module to reload. With DB-based config this becomes moot — but we MUST preserve per-request resolution because a stale cache entry could live up to 30s. Design: call `await getPlatformConfig('crm_bot_enabled', true)` at the TOP of each handler, not memoized.

### Pitfall 3 — First request after deploy has cold cache

Cold-cache DB round-trip adds ~20-50ms to the first request after deploy. Acceptable for CRM bots (latency budget is seconds, not ms). If latency matters later, can pre-warm on module load (fire-and-forget).

### Pitfall 4 — Multi-instance consistency window

Vercel runs multiple concurrent lambda instances. Each has its own in-memory cache. After a flip, different instances may show different values for up to 30s. **Mitigation:** documented in runbook — "post-flip, wait 30s before verifying". For urgent incidents, user can force invalidate via a known code path (POST /admin/invalidate-config with a secret header — deferred to admin UI phase).

### Pitfall 5 — Migration race with deploy

If the new code is deployed BEFORE the table is created, first request throws (table doesn't exist). CLAUDE.md Regla 5 already covers this: migration applied first, code push after. Standard checkpoint.

### Pitfall 6 — Fail-open vs fail-closed behavior

If DB is unreachable (rare), what should `getPlatformConfig('crm_bot_enabled', true)` return?
- Fail-open (return `true` → kill-switch OFF by default): keeps bots running even during DB outage. BUT: if the user had flipped kill-switch to `false` to handle an incident, and DB goes down shortly after, bots resume — undesirable.
- Fail-closed (return `false` → kill-switch ON): safer for incidents but could cause an unnecessary outage if DB has a blip.

**Decision:** fail-open for `crm_bot_enabled` (fallback to `true`). Rationale: DB outages are rare; kill-switch is a soft guard on top of proper rate-limiting + observability. If the user needs a hard kill, they disable API keys (workspace-level, per-key) which doesn't depend on platform_config.

### Pitfall 7 — JSONB coercion bugs

Storing `true` vs `"true"` vs `1` in JSONB: PostgreSQL's JSONB type is strict about JSON types. Insert `'true'::jsonb` → boolean. Insert `'"true"'::jsonb` → string. Our migration must use the correct syntax. `getPlatformConfig<T>` runtime-checks the returned shape and falls back if mismatched.

### Pitfall 8 — Vercel env deletion timing

If user deletes `CRM_BOT_ENABLED` from Vercel BEFORE deploying the refactor, the current live code (still reading process.env) returns undefined → kill-switch activates as side effect. **Order:** deploy refactor first, confirm DB reads work, THEN delete Vercel vars. Document in runbook.

## Recommendations

### Recommendation 1 — Module structure

```
src/lib/domain/platform-config.ts
  ├─ export async function getPlatformConfig<T>(key: string, fallback: T): Promise<T>
  ├─ export function invalidatePlatformConfigCache(key?: string): void
  ├─ PRIVATE: cache = new Map<string, {value, expiresAt}>()
  ├─ PRIVATE: TTL_MS = 30_000
  └─ JSDoc header with Pitfall 4 warning (30s eventual consistency) + Pitfall 6 fallback policy
```

### Recommendation 2 — Migration shape

```sql
CREATE TABLE platform_config (
  key TEXT PRIMARY KEY,
  value JSONB NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT timezone('America/Bogota', NOW())
);

INSERT INTO platform_config (key, value) VALUES
  ('crm_bot_enabled', 'true'::jsonb),
  ('crm_bot_rate_limit_per_min', '50'::jsonb),
  ('crm_bot_alert_from', 'null'::jsonb);

COMMENT ON TABLE platform_config IS 'Platform-level runtime config for CRM bots and future platform features (Phase 44.1). Read via src/lib/domain/platform-config.ts with 30s in-memory cache. No RLS (server-only via createAdminClient).';
```

Additive, no RLS (same pattern as `crm_bot_actions` — server-only), no DROP/ALTER to existing tables.

### Recommendation 3 — Refactor pattern per file

**Routes (reader/propose/confirm):**
```typescript
// BEFORE
if (process.env.CRM_BOT_ENABLED === 'false') return 503...

// AFTER
const enabled = await getPlatformConfig('crm_bot_enabled', true);
if (enabled === false) return 503...
```

**Rate limit resolution in routes:**
```typescript
// BEFORE
const rlResult = rateLimiter.check(workspaceId, 'crm-bot');

// AFTER
const limit = await getPlatformConfig('crm_bot_rate_limit_per_min', 50);
const rlResult = rateLimiter.check(workspaceId, 'crm-bot', { limit });
```

This requires adding an optional `{limit?}` param to `rateLimiter.check`. Minimal, non-breaking.

**`alerts.ts`:**
```typescript
// BEFORE
from: process.env.CRM_BOT_ALERT_FROM || 'onboarding@resend.dev'

// AFTER
const alertFrom = await getPlatformConfig<string | null>('crm_bot_alert_from', null);
from: alertFrom ?? 'onboarding@resend.dev'
```

### Recommendation 4 — Testing

Unit tests for `platform-config.ts`:
- Cache hit within TTL returns memoized value (no DB call)
- Cache miss after TTL re-reads DB
- DB error returns fallback
- `invalidatePlatformConfigCache(key)` clears specific key
- `invalidatePlatformConfigCache()` clears all

Integration tests NOT needed for this refactor — Phase 44's integration tests (reader.test.ts, writer-two-step.test.ts) cover the routes end-to-end and will still pass post-refactor (they test behavior, not config source).

### Recommendation 5 — Don't break Phase 44 Task 6 QA

Phase 44 Plan 44-09 Task 6 is paused on human-verify checkpoint. After this phase lands, the QA procedure changes:
- Old: "set `CRM_BOT_ENABLED=false` in Vercel + redeploy → verify 503"
- New: "`UPDATE platform_config SET value='false'::jsonb WHERE key='crm_bot_enabled'` → wait 30s → verify 503"

Update Plan 44-09 QA checklist in the SUMMARY/LEARNINGS for Phase 44.1 so future auditors understand the transition.

## Known unknowns

- **Supabase JSONB performance** at scale is fine; we have ~3 rows. N/A concern.
- **Next.js/Vercel cold start cache behavior** — each cold start invokes the module fresh, so our in-memory cache is naturally reset. Acceptable.
- **Whether to use Supabase Realtime for instant cache invalidation** — overkill for 3-row config table. Stick with 30s TTL.
