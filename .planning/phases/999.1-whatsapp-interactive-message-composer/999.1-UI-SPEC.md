# Phase 999.1 — UI Design Contract

> Visual and interaction contract for the WhatsApp interactive message composer (botones + lista).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is a frontend phase inside an existing, shipped Next.js 15 + React 19 + Tailwind v4 app.**
> The contract is REUSE-FIRST: every visual and interaction decision is grounded in real in-repo
> files. The executor must NOT invent a new design language — it wires existing primitives
> (`theme-editorial` tokens, shadcn `Dialog`/`Input`/`Textarea`/`Label`/`Button`, `lucide-react`,
> the `whatsapp-bubble.tsx` visual, the `TemplateButton`→Dialog precedent) into 4 narrow surfaces.

---

## Scope (the ONLY 4 areas this spec covers)

1. **Toolbar entry-point button** in `message-input.tsx` — icon, placement, tooltip, 24h-window gating (D-01, D-02).
2. **Builder shell (Dialog)** — type toggle (botones | lista), per-type form fields, inline limit validation display (D-05a), live WhatsApp preview (D-05b), footer send/cancel actions.
3. **Outbound rich bubble render** — `message-bubble.tsx` `case 'interactive'` (D-05c).
4. **Empty / error / loading states** — 360dialog-list-not-supported, window-closed (button hidden), validation errors, send-in-flight + success/failure.

**OUT OF SCOPE** (deferred per CONTEXT.md, do NOT spec): reusable interactive-template library in `/configuracion/whatsapp`; triggering interactives from automations/agents; 360dialog list support; conversation selector inside the composer.

## Spacing Scale

The editorial theme already uses a 4px-based scale expressed in arbitrary Tailwind values. Declared values for this phase (multiples of 4):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Icon-to-label gap inside buttons (`gap-1.5` ≈ 6px is the existing send-button gap; use `gap-1` / `gap-2` for new chips); inline validation-error top margin. |
| sm | 8px | Vertical gap between form fields inside a section; padding of preview button rows (`py-2`). |
| md | 16px | Dialog content horizontal padding; gap between form column and preview column; toolbar button spacing (existing `gap-2` in the input row). |
| lg | 24px | Dialog section breaks (between "tipo", "contenido", "botones/lista", "preview"). |
| xl | 32px | Reserved — only if the Dialog grows a two-column form/preview layout on wide screens. |

**Exceptions (grounded in existing inbox code — keep verbatim, do NOT normalize to the 4px scale):**
- Toolbar icon buttons: `h-10 w-10` (40px) — matches the existing attach/emoji buttons in `message-input.tsx`. This is the touch-target size; keep it.
- Bubble padding: editorial bubble uses `px-[14px] py-[10px]` (14/10px) — non-multiples-of-4 but LOCKED by the existing `message-bubble.tsx` so the new interactive bubble must sit inside the same container unchanged.
- WhatsApp preview bubble: `px-3 py-2` (12/8) and `rounded-lg` from `whatsapp-bubble.tsx` — keep verbatim for WhatsApp fidelity.

## Typography

The phase has two typographic zones. **Zone A = editorial UI chrome** (Dialog labels, helper text, validation errors, footer buttons). **Zone B = WhatsApp preview/bubble** (must look like real WhatsApp, escapes editorial fonts).

| Role | Size | Weight | Line Height | Zone / token |
|------|------|--------|-------------|--------------|
| Body (form inputs, helper text) | 14px | 400 regular | 1.5 | Zone A — `text-[14px]`, `--font-sans`. Mirrors the existing inbox input (`text-[14px] [font-family:var(--font-sans)]`). |
| Label (field labels, type toggle) | 13px | 600 semibold | 1.4 | Zone A — `--font-sans`. Matches send-button label weight (`font-semibold text-[13px]`). |
| Validation error / counter | 11px | 500 medium | 1.4 | Zone A — `text-[11px]`, `--font-sans`; error color = `--rubric-2` / `text-destructive`. |
| Dialog heading | 16px | 600 semibold | 1.2 | Zone A — shadcn `DialogTitle` default; `--font-sans`. |
| Preview/bubble body | 14-15px | 400 regular | 1.5 | Zone B — `text-sm whitespace-pre-wrap break-words` from `whatsapp-bubble.tsx`. Native black/white text. |
| Preview/bubble header | 14px | 600 semibold | 1.4 | Zone B — `font-semibold text-sm` from `whatsapp-bubble.tsx`. |
| Preview/bubble footer + button labels | 13px | 400-500 | 1.4 | Zone B — `text-xs opacity-70` for footer; interactive buttons render as WhatsApp-style action rows (see Visuals). |

**Total declared sizes: 4 (14, 13, 11, 16)** — within the 3-4 limit. **Weights: 2 (regular 400 + semibold 600)**, with one medium (500) reused only for the counter/timestamp exactly as the existing inbox does (not a new weight — it is the established `--font-mono`/counter weight already shipped).

## Interaction Contract (per scope area)

### Area 1 — Toolbar entry-point button

- **Placement:** Inside `message-input.tsx`, in the OPEN-window return branch only (the toolbar `div` ~L416-437), positioned next to the attach (Paperclip) and emoji (Smile) ghost buttons, BEFORE the text input. It must NOT appear in the closed-window branch (L291-333), which only offers `TemplateButton` (D-02 / RESEARCH Pitfall 2).
- **Shape:** `<Button variant="ghost" size="icon" className="h-10 w-10 flex-shrink-0">` — identical to attach/emoji.
- **Icon:** `lucide-react` `List` (the interactive message is fundamentally a list/buttons builder; `List` reads clearly and is not already used in the toolbar). Size `h-5 w-5`.
- **Tooltip / `title`:** `"Mensaje interactivo"`.
- **Disabled rule (D-02):** The button only renders when `isWindowOpen === true`. There is no separate disabled visual — the 24h window IS the gate (closed window → the whole composer is replaced by the warning strip + template button, so the interactive button simply does not exist in that branch). Additionally disable (`disabled={isLoading}`) while a send is in flight, matching the emoji button.
- **`aria-label`:** `"Abrir compositor de mensaje interactivo"`.

### Area 2 — Builder shell

- **Container: Dialog, NOT Popover** (Claude's discretion, D-05/UI — JUSTIFIED): the list builder needs vertical room for sections + rows + a live preview + inline errors. The emoji Popover is fine for a compact widget, but the closest in-repo precedent for "toolbar button → builder modal with preview + send" is `TemplateButton`→`TemplateSendModal`, which is Dialog-based. Mirror it: `Dialog` + `DialogContent` (`max-w-2xl` to fit form + preview side-by-side on desktop, single-column stacked on narrow) + `DialogHeader` (`DialogTitle` = "Mensaje interactivo", `DialogDescription` = "Arma botones o una lista y envíalos a esta conversación.") + a `ScrollArea` body + `DialogFooter`.
- **Type toggle (botones | lista):** A 2-option segmented control at the top of the body. Use shadcn `ToggleGroup` (type="single", value-bound) or `Tabs` — prefer `Tabs` with two `TabsTrigger` ("Botones", "Lista") so the form below swaps cleanly. Default = "Botones" (simplest, most common). Label weight 600, 13px.
- **Form fields — shared (both types):**
  - `header` (opcional) — single-line `Input`, label "Encabezado (opcional)", live char counter `n/60`.
  - `body` (obligatorio) — multiline `Textarea`, label "Cuerpo del mensaje", counter `n/1024`, min 1 char.
  - `footer` (opcional) — single-line `Input`, label "Pie de página (opcional)", counter `n/60`.
- **Form fields — Botones:**
  - 1-3 reply buttons. Each row = a single-line `Input` for the button title (label "Botón N", counter `n/20`) + a remove (`X` icon) button when count > 1.
  - "Agregar botón" `Button variant="outline" size="sm"` — hidden/disabled when count === 3.
  - **IDs auto-generated, never operator-typed** (Claude's discretion D-05/id — JUSTIFIED): generate stable unique ids as `btn_${index}` (or short `crypto.randomUUID().slice(0,8)`), enforce uniqueness in the validation helper. The operator never sees or edits ids.
- **Form fields — Lista:**
  - `buttonLabel` (obligatorio) — the list-open button text, single-line `Input`, label "Texto del botón de lista", counter `n/20`.
  - 1-10 sections. Each section = an optional section-title `Input` (label "Sección N título (opcional)", counter `n/24`) + 1-N rows.
  - Each row = row-title `Input` (obligatorio, counter `n/24`) + row-description `Input` (opcional, counter `n/72`) + remove `X`.
  - "Agregar fila" and "Agregar sección" `Button variant="outline" size="sm"`.
  - Row ids auto-generated `row_${sectionIndex}_${rowIndex}` (or short uuid), unique across ALL sections.
- **Inline validation (D-05a) — display contract:** Use the `interactive-limits.ts` helper (RESEARCH §Code Examples) — the FULL Meta limit set, INCLUDING **total rows across all sections ≤ 10** (RESEARCH Pitfall 1). Display rules:
  - Each field shows a live char counter; when over limit, the counter turns `text-destructive` (rubric-2) and a `text-[11px] text-destructive` message appears below the field (e.g. "Máx 20 caracteres").
  - Aggregate errors (e.g. "Máx 10 filas en total", "Al menos 1 botón", "IDs de fila duplicados") render in a compact list above the footer.
  - The "Enviar" button is `disabled` whenever the validation helper returns any error.
- **Footer actions (`DialogFooter`):**
  - Primary: `"Enviar mensaje"` — `Button` styled as the editorial primary (`bg-[var(--ink-1)] text-[var(--paper-0)]`, ink fill, matching the inbox send button), with a `Loader2` spinner + label `"Enviando…"` while in flight, `disabled` when validation fails or sending.
  - Secondary: `"Cancelar"` — `Button variant="ghost"` / `variant="outline"`, closes the Dialog without sending. No destructive confirmation needed (nothing is persisted until send).

### Area 3 — Outbound rich bubble render (D-05c)

- Replace the `case 'interactive'` placeholder (`message-bubble.tsx` L133-140) with a rich render that consumes the stored `content` JSONB (`{ interactiveType, body, header?, footer?, buttons? | buttonLabel?+sections? }`).
- **Reuse the SAME pure render component used by the live preview** (build one `interactive-bubble.tsx`, consumed by both the preview and this case — single source of truth, RESEARCH Pattern 4). This guarantees the operator's preview and the sent bubble are pixel-identical.
- **Visual:** header (semibold) → body (`whitespace-pre-wrap break-words`) → footer (`text-xs opacity-70`) → then the interactive affordance:
  - **Botones:** a vertical stack of WhatsApp-style reply buttons — each a full-width row, top-border-separated, centered label, no real action (display-only echo of what was sent). Mirror WhatsApp's button look: `text-[#00a5f4] dark:text-[#53bdeb]` link-blue label, `border-t` separators, `py-2 text-sm font-medium text-center`.
  - **Lista:** a single full-width "list-open" row showing the `buttonLabel` with a list glyph (lucide `List` `h-4 w-4`), same WhatsApp link-blue + `border-t` treatment. (Sections/rows do not need to be expanded in the bubble — WhatsApp itself shows only the button until tapped; an optional muted summary line "N opciones" is acceptable.)
- The rich content sits inside the existing editorial bubble container (L201-220) unchanged — do NOT alter the bubble shell, padding, timestamp, or status-icon row. Only the inner content region changes.
- **XSS:** plain text only, React auto-escaping, NO `dangerouslySetInnerHTML` (matches `whatsapp-bubble.tsx`).

### Area 4 — Empty / error / loading states

| State | Trigger | UI contract |
|-------|---------|-------------|
| Window closed | `isWindowOpen === false` | Interactive button not rendered at all (lives only in the open-window branch). No separate message needed — the existing "Ventana de 24h cerrada" strip already explains the state. |
| Empty builder | Dialog just opened | Default = Botones type, 1 empty button row, empty body. Live preview shows the `whatsapp-bubble.tsx` placeholder `"(escribe el body...)"` until the operator types. |
| Validation error | Helper returns errors | Per-field counters + inline messages in `text-destructive`; aggregate list above footer; "Enviar" disabled. (See Area 2.) |
| 360dialog list not supported | Server action returns `error: 'lista no soportada en 360dialog (legacy)'` | Surface via `toast.error` with a FRIENDLY, provider-agnostic message (do NOT leak provider mechanics): **"Las listas no están disponibles en esta conexión de WhatsApp. Usa botones."** Keep the Dialog open so the operator can switch to Botones. (Provider stays domain-only — Regla 3 — so the client cannot pre-hide the list tab; attempt-and-error is the correct pattern, RESEARCH Pitfall 3 / Open Q1.) |
| Send in flight | "Enviar" clicked, awaiting action | Footer button shows `Loader2` spinner + "Enviando…", button disabled, form disabled. |
| Send success | Action returns success | `toast.success("Mensaje interactivo enviado")`, close Dialog, call `onSend?.()` so the inbox refetches/echoes the new outbound bubble (mirror the existing send paths). |
| Send failure (other) | Action returns a non-list error | `toast.error(result.error ?? "Error al enviar mensaje interactivo")`, keep Dialog open, re-enable form. Optionally offer a "Reintentar" toast action like the text-send path. |
